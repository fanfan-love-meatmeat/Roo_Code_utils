# 风险评估 — Roo_Code_utils MCP Server

> **审计时间**: 2026-05-26 16:40 CST
> **审计版本**: v1.0 (初次审查，基于《项目现状梳理》v1.0)
> **风险等级说明**: P0-致命 / P1-高危 / P2-隐患

---

## 1. 工业级水位判定

基于联网查证与工业标准对标，当前 `Roo_Code_utils` 的整体架构水位评估如下：

| 维度 | 水位 | 评级 | 说明 |
|------|------|------|------|
| 核心算法能力 | **领先** | ⭐⭐⭐⭐⭐ | MTEF 纯 Python 双轨引擎在开源生态中无平替方案，对标的是商业产品 GrindEQ ($99) |
| 协议层适配 | **达标** | ⭐⭐⭐ | FastMCP 选型正确，但仅使用最基础的 stdio + 装饰器模式 |
| 安全防护 | **严重不足** | ⭐ | 无输入验证、无路径遍历防护、无认证授权、无错误脱敏 |
| 可运维性 | **严重不足** | ⭐ | 无结构化日志、无健康检查、无监控指标、无版本锁定 |
| 代码工程质量 | **中下** | ⭐⭐ | 存在重复代码块、死代码、未同步文档、非确定性解析逻辑 |
| 可扩展性 | **低** | ⭐⭐ | 单文件脚本架构、硬编码路径、无配置中心、无插件机制 |

**综合判定**: 项目在**核心算法维度达到工业领先水平**，但在**安全、运维、工程质量三个维度存在显著的技术债务**。以当前状态直接推向生产环境（即使仅限单机 IDE 集成场景）仍面临多个可被触发的 P0/P1 级风险。

## 2. 核心风险点排序

### [P0-致命] 风险 — 可导致系统崩溃、数据损坏或安全漏洞

#### P0-1: MCP 工具无输入验证与路径遍历防护

- **位置**: `server.py:17-91` (4 个 MCP tool 装饰器函数)
- **触发条件**: AI Agent 或用户传入恶意构造的文件路径
- **攻击向量**:
  - 路径遍历: `../../etc/passwd` 或 `..\\..\\Windows\\System32\\config\\SAM`
  - 空路径 / 非文件路径 / 目录路径 / 设备文件路径
  - 超大路径名 (>260 字符 Win32 限制)
- **后果**: 
  - MCP 服务进程可被诱导读取/写入任意文件系统位置
  - 结合 OLE 解析的二进制读取，可能触发缓冲区级漏洞
  - 参考: CVE-2025-49596 (MCP Inspector RCE, CVSS 9.4)
- **工业参考**: [MCP Security Best Practices (StackHawk)](https://www.stackhawk.com/blog/mcp-server-security-best-practices)、[Snyk: Preventing Path Traversal in MCP](https://snyk.io/articles/preventing-path-traversal-vulnerabilities-in-mcp-server-function-handlers)

#### P0-2: olefile 二进制解析安全边界缺失

- **位置**: `tools/docx_to_md.py:117-133` (`_build_formula_map` 中 olefile 解包)
- **触发条件**: 处理恶意构造的 DOCX 文件（包含畸形 OLE 对象）
- **攻击向量**:
  - `olefile.isOleFile()` 存在已知的恶意 OLE2 文件崩溃/DoS 漏洞
  - `ole.openstream("Equation Native").read()` 对返回的数据量无上限控制，可导致内存耗尽 (OOM)
  - `raw[28:]` 切片假设所有 oleObject.bin 都有 28B 头部，畸形文件可触发 IndexError 崩溃
  - 上游 `zipfile.ZipFile` 存在 Zip Slip 漏洞风险（未对 zip 条目名做路径规范检查）
- **后果**: 服务进程崩溃 (DoS)、内存耗尽 (OOM)、潜在代码执行（若 olefile 存在未修复漏洞）
- **工业参考**: olefile 自 2018 年停止活跃维护，[Python Security Best Practices](https://www.aikido.dev/blog/python-security-vulnerabilities)

#### P0-3: 无 Structured Error Handling — 错误信息裸奔泄露

- **位置**: `server.py:27-31, 46-47, 62-63, 89-91`
- **触发条件**: 任何内部异常
- **现状**: 所有异常均通过 `return f"xxx失败: {str(e)}"` 直接返回，无 `mask_error_details`、无异常分类
- **后果**:
  - 内部路径、文件系统结构、依赖版本信息通过异常消息泄露给 AI Agent 上下文
  - AI Agent 可能基于泄露信息进行进一步危险操作
  - 与 FastMCP 官方推荐的 `mask_error_details=True` 和 `ToolError` 机制背道而驰
- **工业参考**: [FastMCP Error Handling Middleware](https://gofastmcp.com/python-sdk/fastmcp-server-middleware-error_handling)

### [P1-高危] 风险 — 可导致数据丢失、静默错误或退化行为

#### P1-1: 大文件处理无资源限制 — 物理资源耗尽点

- **位置**: 全局 (`docx_to_md.py`, `pdf_to_text.py`, `pdf_to_images.py`)
- **触发条件**: 处理大体积 DOCX (>100MB) 或 PDF (>500 页)
- **具体风险**:
  - `Document(docx_path)` 全量加载到内存，无流式处理
  - `_build_image_map` 中所有图片全量解压到内存
  - PDF 转图片时逐页 `get_pixmap` 无内存回收机制
  - 无文件大小上限检查、无超时控制
- **后果**: 单次处理可耗尽系统内存导致进程被 OS Kill，且无优雅降级提示

#### P1-2: 文件写入竞态条件与原子性缺失

- **位置**: `docx_to_md.py:183-189`、`pdf_to_images.py:28-30`
- **触发条件**: 并发调用（即使当前 stdio 单进程看似安全，若未来切换到 HTTP/SSE 模式则暴露）
- **具体风险**:
  - `_build_image_map` 中同一文件被连续写入两次（代码重复 bug，lines 183-189）
  - `pdf_to_images` 中 `page_*.png` 无临时文件 + 原子重命名机制
  - 图片去重逻辑 (line 178-181) 在高并发下存在 TOCTOU 竞态
- **后果**: 文件截断、输出损坏、磁盘 I/O 浪费

#### P1-3: XSLT 文件外部依赖 — Single Point of Failure

- **位置**: `docx_to_md.py:45-46, 56-70`
- **触发条件**: `tools/xslt/` 目录下任一 `.xsl` 文件缺失或被误删
- **现状**: 
  - 在模块导入时通过全局变量 + lazy init 加载，加载失败直接抛出 `FileNotFoundError`
  - 8 个 XSLT 文件存在文件系统耦合，无内置 fallback 或嵌入方案
  - 若部署路径变更或打包分发，OMML 公式转换将全面崩溃
- **后果**: 所有包含 OMML 公式的 DOCX 文档将转换失败，且错误信息不友好

#### P1-4: 内存泄漏与资源未释放

- **位置**: `docx_to_md.py`、`pdf_to_text.py`、`pdf_to_images.py`
- **触发条件**: 连续多次调用（长生命周期 MCP 服务进程）
- **具体风险**:
  - `docx_to_md.py` 中用 `zipfile.ZipFile` 打开 DOCX 但未在异常路径中显式关闭
  - `pdf_to_text.py` 和 `pdf_to_images.py` 中有 `doc.close()` 但异常路径可能跳过
  - `lxml.etree.XSLT` 转换器通过模块级全局变量缓存，但变换后的结果树未显式释放
- **后果**: 长时间运行后内存持续增长，最终 OOM

#### P1-5: 非确定性解析 — MTEF 自校准算法

- **位置**: `mtef_parser.py:300-346` (`_consume_dimension_array`)
- **触发条件**: 非标准 MathType 生成的 MTEF 数据或格式损坏的 oleObject.bin
- **现状**: EQN_PREFS nibble 尺寸数组的 style_count 边界使用启发式自校准算法 — 通过"后一字节是否为有效 MTEF tag"来试探性定位，并在失败时打印 warning 继续执行
- **后果**: 解析游标可能在错误位置继续解析，产生伪 LaTeX 输出或静默吞字符，且下游无法感知数据损坏

### [P2-隐患] 风险 — 代码质量、维护性与渐进退化

#### P2-1: 代码重复与死代码积累

| 文件 | 问题 | 行号 |
|------|------|------|
| `docx_to_md.py` | `_build_image_map` — 同一文件写入代码重复执行两次 | 183-189 |
| `docx_to_md.py` | `_parse_paragraph_children` — AlternateContent 处理逻辑重复两段 | 439-479 |
| `docx_to_md.py` | `_cleanup_stale_files` — 定义了但从未调用 | 236-243 |
| `docx_to_md.py` | `_split_image_dirs` — 定义了但从未调用 | 196-233 |
| `docx_to_md.py` | 第二段 AlternateContent 中使用了 `OCR_INLINE`/`OCR_BLOCK` 标签 — OCR 管线已移除 | 474-476 |

#### P2-2: 文档-代码不一致

- **位置**: `environment_backup.md:14-15`、`environment_backup.md:27`
- **问题**: 仍包含 `pip install pix2tex torch` 命令和 "DOCX→MD 公式 OCR (MathType→LaTeX)" 功能描述，但该管线已于 2026-05-25 完全移除
- **后果**: 新成员按文档搭建环境时会安装不必要的 GB 级依赖（torch）

#### P2-3: 硬编码路径与缺乏配置抽象

- **位置**: 多处
- **问题**:
  - MCP 配置中的 Python 解释器路径硬编码为 `C:\Users\98514\.conda\envs\roo_mcp\python.exe`
  - 输出目录逻辑硬编码判断 `os.path.basename(src_dir) == "raw"`
  - 无环境变量 / 配置文件驱动的参数注入
- **后果**: 项目迁移到其他机器或用户时需手动修改多处硬编码

#### P2-4: 依赖版本未锁定

- **问题**: 无 `requirements.txt`、`pyproject.toml`、`Pipfile.lock` 或 `conda-lock` 文件
- **后果**: 依赖项 (`mcp`, `pymupdf`, `python-docx`, `lxml`, `Pillow`, `olefile`) 的版本未锁定，随时间推移可能出现 API 兼容性破坏

#### P2-5: MD → DOCX 转换能力边界未声明

- **位置**: `md_to_docx.py`
- **问题**:
  - 不支持嵌套列表、代码块 (` ``` `)、内联代码 (`` ` ``)、LaTeX 公式
  - 表格解析假设严格规范格式（跳过分隔符行），对不规则表格静默丢弃数据
  - Emoji 替换逻辑 (`✅→√`, `✓→√`) 硬编码且不完整
- **后果**: 用户（AI Agent）对能力边界无感知，可能产生静默内容丢失

#### P2-6: .gitignore 过于激进

- **位置**: `.gitignore:9`
- **问题**: `raw_md/` 被整体忽略，但该目录是工具的核心输出区，若用户期望版本控制输出则无法追踪
- **后果**: 输出文件无法通过 git 追溯到对应的输入版本

## 3. 务实止血预案 (仅限架构层面)

### P0 止血方案

| 风险 | 止血建议 |
|------|---------|
| **P0-1 输入验证缺失** | 在 `server.py` 的 4 个 tool 函数入口添加统一的 `validate_file_path` 中间件：<br>1. 使用 `os.path.realpath` 解析并校验路径在允许的基础目录内（白名单目录约束）<br>2. 校验文件扩展名白名单 (`.docx`/`.pdf`/`.md`)<br>3. 校验文件大小上限（如 DOCX < 100MB, PDF < 200MB）<br>4. 拒绝包含 `..` 的相对路径组件 |
| **P0-2 olefile 安全边界** | 1. 在 `_build_formula_map` 中对 `oleObject.bin` 添加大小上限检查（如 10MB）<br>2. 对 `zipfile.ZipFile` 的 `namelist` 逐项做 `os.path.basename` 消毒（防 Zip Slip）<br>3. 考虑评估 `olefile` 的替代方案或 fork 修复已知漏洞 |
| **P0-3 错误处理** | 1. 在 `FastMCP` 构造函数中启用 `mask_error_details=True`<br>2. 对可预期的异常（`FileNotFoundError`、`PermissionError`、`ValueError`）使用 `ToolError` 返回可操作的提示消息<br>3. 不可预期的内部异常通过日志记录后返回通用错误码 |

### P1 止血方案

| 风险 | 止血建议 |
|------|---------|
| **P1-1 资源限制** | 1. 在工具入口添加文件大小和页数上限检查<br>2. 对图片提取添加总量上限和单图片大小上限<br>3. 在 `pdf_to_images` 中添加逐页的显式内存回收（`pix = None` / `del pix`） |
| **P1-2 竞态条件** | 1. 修复 `_build_image_map` 中重复写入的代码缺陷<br>2. 所有文件写入改用 `tempfile` + `os.replace`（原子写入） |
| **P1-3 XSLT SPOF** | 1. 在 `server.py` 启动时做 XSLT 文件可用性预检（pre-flight check），缺失时拒绝启动并记录明确错误<br>2. 考虑将 XSLT 内容嵌入 Python 模块（`importlib.resources`）或 base64 编码内联 |
| **P1-4 资源泄漏** | 1. 所有文件操作统一使用 `with` 上下文管理器或 `try/finally` 确保关闭<br>2. 周期性调用 `gc.collect()` 或利用 Python 引用计数自然回收<br>3. 考虑 MCP 服务进程的定期重启策略（如处理 N 个文档后自动退出由客户端重拉） |
| **P1-5 非确定性解析** | 1. 对 `_consume_dimension_array` 的自校准增加更严格的边界校验（如校验解析后的 style_count 与其他字段的一致性）<br>2. 在校准失败时标记整个公式为 `[公式: 结构异常]` 而非继续尝试解析 |

### P2 止血方案

| 风险 | 止血建议 |
|------|---------|
| **P2-1 死代码** | 删除所有未调用函数（`_cleanup_stale_files`、`_split_image_dirs`）和重复代码块，移除残留的 OCR 标签引用 |
| **P2-2 文档同步** | 更新 `environment_backup.md` 移除 OCR 管线和 pix2tex/torch 依赖，对齐实际依赖清单 |
| **P2-4 依赖锁定** | 创建 `requirements.txt` 并固定所有直接依赖的版本号，建议补充 `conda environment.yml` |
| **P2-6 .gitignore** | 将 `raw_md/` 从 `.gitignore` 移除或改为仅忽略 `raw_md/*/figures/` |
