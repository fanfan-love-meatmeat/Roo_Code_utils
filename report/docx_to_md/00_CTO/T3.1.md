### 1. 乱码根因深度剖析 (Technical Dissection)

为什么指针完全对齐了，吐出来的却依然是乱码？
因为目前的 `CHAR` 解析算子陷入了**“纯文本盲区”**。

在 MTEF 协议中，当状态机步入 `CHAR` (0x02) 记录时，它读取到的二进制数据并不是标准的 UTF-8 或 Unicode 编码，而是 MathType 内部独立的**字符偏移索引（Character Code）**。
* **致命误区**：目前的代码在读取到 16 位变长字符值后，直接简单粗暴地调用了 Python 原生的 `chr(char_val)`。
* **走火入魔**：由于没有经过字体映射表的拦截，这些在 MathType 私有编码集（如 MTCode）中的数学符号索引值，被强行强制转换成了 Unicode 字符，从而映射到了 CJK 统一表意文字扩展区或私有区（PUA），变成了我们看到的 `뢄`、`햄`、`碃`。

**典型符号对照推演：**
* 文本中的 `$뢄{}$` 实际对应的是云台控制算法中的 **俯仰角 ($\theta_p$)**。
* 文本中的 `$햄{}$` 实际对应的是云台控制算法中的 **偏航角 ($\theta_y$)**。
* 文本中的 `$撃$` 和 `$澃$` 则紧密关联深度相机解算出的 **目标深度值 ($Z$)** 与 **车辆前进方向投影距离 ($d_{proj}$)**。

### 2. Task 3.1 核心重构方针 (Engineering Directives)

团队不需要在此报告中寻找现成的代码，我要求模块负责人紧扣以下三层架构逻辑，手写完成 `mtef_parser.py` 的解耦与升级：

#### 🛠️ 指令一：激活 Typeface 联动，构建字源映射机制 (Typeface-Driven Mapping)
每个 `CHAR` 记录在头部都声明了一个 `typeface`（字体索引）字节。这个字节不是摆设，它是解锁字符集真名的钥匙。
* **实现要求**：重构 `_read_char_value()` 方法。必须首先检索该 `CHAR` 所属的 `typeface` 属性，识别其对应的是 `Symbol`、`MT Extra` 还是 `MTCode`（定义在 `FONT_DEF` 阶段）。
* **映射路由**：根据不同的字体通道，将读取到的二进制字符值（Character Code）路由到对应的静态映射表中，将其精准转译为标准的 LaTeX 宏命令（例如：将特定的十六进制值拦截并映射为 `\theta`、`\Delta`、`\times`），严禁直接放行进入 `chr()`。

#### 🛠️ 指令二：实现私有区 (PUA) 符号的启发式对齐 (PUA Heuristics)
鉴于部分手敲公式可能由于历史编辑器的版本更迭导致字体索引丢失。
* **实现要求**：设计一个启发式降级算子。当发现映射未命中且 `char_val` 落在 Unicode 私有区（`0xE000 - 0xF8FF`）或特定的生僻 CJK 区域时，根据常用数学符号的二进制高低位特征进行动态嗅探，确保至少能回退到合理的希腊字母或排版操作符，拒绝输出破坏 Markdown 语义的“方块字”或生僻字。

#### 🛠️ 指令三：强化状态机对齐断言与日志捕获 (State Alignment Assertion)
* **实现要求**：在 `_parse_line` 的迭代管线中，一旦检测到映射表完全失效，必须将当前公式的完整十六进制上下文（Context Hex Dump）以及冲突的 `typeface` 编号以 Debug 日志形式抛出，为接下来的符号表微调留出精确的口子。
