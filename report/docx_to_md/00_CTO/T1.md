# 🚀 架构决策与技术路线转移报告：弃用闭源 DLL，全面转向纯 Python MTEF 双轨解析引擎 (v3.2)

## 1. 决议概述
鉴于最新 MVP 测试的探雷结果，**即日起正式废弃基于 `MT6.dll` 挂载的公式解析方案，全面转向自主可控的纯 Python MTEF 二进制解析架构 (v3.2)。** 在此，我要特别肯定团队在 Phase 1 阶段执行的极简 MVP 连通性测试。这次极其规范的“Fail Fast（快速试错）”为我们扫清了一个隐蔽极深的底层架构地雷，避免了后期大规模重构的灾难，是一次极具工程价值的排险行动。

---

## 2. MVP 测试复盘与风控拦截

在 64 位环境下的测试表明，尽管 API 握手成功，但核心转换引擎 `MTXFormEqn` 始终返回 `-9999`，且 `MTAPIVersion` 返回 `0x0`。
* **根因确证**：MathType 7.x 转向 SaaS 订阅制后，其底层 DLL 已被重构为依赖 `MathType.exe` 宿主进程的 IPC 通信组件。在无头（Headless）挂载环境下，缺少 GUI 进程与授权 Token 的上下文，DLL 会从底层切断转换管线。
* **风控底线**：在 Agri-CPS 系统的高密集推演和无头 Agent（如 Roo Code）的自动化工作流中，我们绝不能将核心数据管线建立在一个被阉割、且强依赖外部 GUI 状态机的闭源商业黑盒之上。

在回退路线的评估上，我们**坚决否决**回归 v2.3 的视觉 OCR 管线（无法满足 100% 准确率红线），也**坚决否决** Word COM/pywin32 自动化方案（高延迟与进程挂起风险将摧毁 Agent 推演效率）。

**数据本源就在我们手中，唯一正途是直面二进制协议。**

## 3. v3.2 架构破局点：MTEF 元数据发现与双轨火控

通过对解包后 281 字节 `Equation Native` (MTEF v5) 数据流的 Dump 审查，我们发现了一个可以直接利用的底层机制：**如果公式是由 TeX 语法输入的，MathType 会将原始的 LaTeX 代码明文作为元数据（Metadata）储存在 MTEF 二进制树的头部。**

基于此，我们将构建一个**零外部依赖、极低开销**的纯 Python 双轨解析引擎：

### 🛤️ Track 1: Fast-Path（极速斩首层 - 优先执行）
* **逻辑**：绕过复杂的二进制反序列化，直接对 `olefile` 提取出的 MTEF 字节流进行特征码切片。
* **实现**：定位二进制特征 `b'TeX Input Language\x00'`，向后提取直到遇到下一个 `\x00` 终止符。
* **收益**：**微秒级（μs）提取**，瞬间剥离完美无损的 LaTeX 明文。覆盖绝大多数通过学术规范输入的公式。

### 🛤️ Track 2: Slow-Path（深度解析层 - 兜底回退）
* **逻辑**：对于少数纯通过 MathType 面板鼠标点击生成的公式（缺少 TeX 元数据），触发纯 Python 编写的 MTEF v5 解析器。
* **实现**：根据公开的 MTEF v5 规范，从 `0x88` 偏移量起，递归解析 `SIZE`, `LINE`, `CHAR`, `TMPL` 等二进制 Tag，动态拼装 LaTeX 字符串。
* **收益**：**100% 场景覆盖**。彻底摆脱对本地 Windows 环境和商业软件的依赖，实现全平台的降维打击。
