# Debug Task 02: 拦截 Agent 路径幻觉与基础输入防御 (Fix P0-1)

## 1. 任务背景
在当前架构下，无视觉 Agent（特别是配置为 V2.4 Engine 运行时）在处理多步复杂指令时，偶尔会产生上下文幻觉，构造出物理磁盘上并不存在的文件路径。目前系统缺乏最外层拦截，导致这些非法路径被直接透传给底层的 OLE 或 XML 解析引擎，从而引发未捕获的严重异常，使得整个 MCP 服务进程崩溃断联。我们需要通过“快速失败（Fail-Fast）”机制来保障 Agent 对话链路的稳定性。

## 2. 核心目标
在 `server.py` 暴露给 Agent 的工具入口处，建立一层极其轻量的防御拦截。通过标准化的错误反馈，打断 Agent 的错误尝试并引导其自我修正，避免底层崩溃。

## 3. 验收标准 (Acceptance Criteria)
* **存在性校验 (Existence Check)**: 在任何文件 I/O 发生前，必须校验 Agent 传入的路径对应的文件是否真实存在。
* **类型校验 (Extension Check)**: 根据具体工具的职责，校验文件后缀名是否在允许的白名单内（如仅允许 `.docx`、`.pdf` 或 `.md`）。
* **标准化阻断反馈 (Standardized Interruption)**: 触发拦截时，**必须**通过 FastMCP 的 `ToolError` 机制返回语义清晰、可指导 Agent 纠错的自然语言提示（例如：“目标文件不存在，请先检查路径是否正确”）。严禁直接引发系统底层异常或将 Python 错误堆栈暴露给大模型上下文。
* **全面覆盖**: 该前置校验逻辑需覆盖 `server.py` 中所有注册为 `@mcp.tool` 的函数入口。

## 4. 边界与约束 (Scope & Constraints)
* **无需过度防御**: 鉴于该系统主要用于本地核心手稿流转和个人使用，当前阶段**不需要**实现复杂的沙箱隔离，也**不需要**编写繁琐的正则表达式来防御蓄意的安全攻击（如高级别路径穿越漏洞）。
* **聚焦上下文净化**: 重点在于优化系统与 Agent 之间的通信质量。拦截逻辑越轻量越好，核心指标是确保服务不挂，且 Agent 能根据错误提示自行调整路径。
