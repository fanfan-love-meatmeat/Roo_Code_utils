## 3. 补充优化自检

以下为技术推演中建议但 CTO 设计未明确要求的补充项：

| 补充项 | 实施状态 | 理由 |
|--------|---------|------|
| `mask_error_details=True` | ✅ 已实施 | 无此参数则内部异常堆栈仍泄露；与 ToolError 配合形成完整的错误脱敏/透传双层机制 |
| 文件大小上限 (100MB) | ✅ 已实施 | 防止大文件 OOM；100MB 对学术论文场景足够宽松 |
| 隐性路径扩容修复（`except Exception: raise` 替代 `return f"失败: {str(e)}"`） | ✅ 已实施 | 旧模式导致 MCP 框架将错误视为成功响应（`isError: false`），新模式正确终止为错误响应 |

## 4. 未变更项与理由

| 未变更 | 理由 |
|--------|------|
| 可选输出参数 (`docx_path`, `output_dir`, `output_md_path`, `image_dir`) 的校验 | 输出路径无需存在性前置检查；父目录权限校验超出本次范围，属于 P1-2 (竞态/原子性) 任务 |
| 路径遍历防御 (`os.path.commonpath`) | CTO 明确排除，stdio 单机场景威胁模型不同 |
| 底层引擎 (`tools/*.py`) 的修改 | 保持防御层与业务层分离；本次仅在 MCP 入口层做 Fail-Fast |
| `_validate_input_file` 对工具名不存在的兜底 | `_ALLOWED_EXTENSIONS.get(tool_name, [])` 返回空列表则跳过扩展名校验，行为安全 |
