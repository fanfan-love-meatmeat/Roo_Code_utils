### 1.2 FastMCP 异常模型深度分析

FastMCP 的异常处理模型存在**两层机制**，这是理解 CTO 指令正确性的关键：

```
┌─────────────────────────────────────────────────────────┐
│              异常处理层级                                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  mask_error_details = False (默认)                       │
│  ┌──────────────┐    ┌──────────────┐                   │
│  │ ToolError    │───▶│  完整消息    │──▶ LLM 上下文     │
│  │ ValueError   │───▶│  完整消息    │──▶ LLM 上下文     │
│  │ Exception    │───▶│  完整 trace  │──▶ LLM 上下文 ⚠   │
│  └──────────────┘    └──────────────┘                   │
│                                                         │
│  mask_error_details = True                               │
│  ┌──────────────┐    ┌──────────────┐                   │
│  │ ToolError    │───▶│  完整消息    │──▶ LLM 上下文 ✅   │
│  │ ValueError   │───▶│  "Internal   │──▶ LLM 上下文 ❌   │
│  │ Exception    │───▶│   error"     │──▶ LLM 上下文 ❌   │
│  └──────────────┘    └──────────────┘                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**关键结论**（来自 FastMCP 官方文档）：
> "Error messages from ToolError are **always** sent to clients, regardless of mask_error_details setting."

**当前 server.py 的现状**：
- `mask_error_details` 未启用（使用默认值 `False`）
- 所有异常通过 `return f"xxx失败: {str(e)}"` 返回**字符串**（不是 raise 异常）
- 这意味着即使内部发生 `FileNotFoundError`，它被 `except Exception` 捕获并转为成功返回的字符串，**不会被 FastMCP 框架感知为错误**

**CTO 指令 `raise ToolError(...)` 的关键变化**：
- 将验证失败的"返回值"改为"异常抛出"，使得 FastMCP 框架能够正确识别为 tool error
- 这比当前 `return "失败: ..."` 模式更符合 MCP 协议的语义（error response vs success response）
- 后续若启用 `mask_error_details=True`，ToolError 消息仍能完整透传，不受影响

### 1.3 CTO 设计逐项校验

| CTO 验收标准 | 与业界最佳实践对比 | 校验结果 |
|-------------|------------------|---------|
| **存在性校验 (os.path.exists)** | Firecrawl 生产指南示例：`if not os.path.exists(expanded_path): return False, f"File not found: {file_path}"` | ✅ 完全一致 |
| **类型校验 (扩展名白名单)** | BigDataBoutique 7 大错误之 #2："Validate all inputs... treat LLM-generated strings as untrusted" | ✅ 完全一致 |
| **ToolError 标准化阻断** | FastMCP 官方推荐：`raise ToolError("Division by zero is not allowed.")` + `mask_error_details=True` | ✅ 核心机制正确（见 1.4 补充建议） |
| **严禁暴露 Python 堆栈** | MCP Security Best Practices (Enkrypt AI): "mask internal error details for security" | ✅ 方向正确，但需 `mask_error_details=True` 才能完全实现 |
| **全面覆盖 4 个 @mcp.tool** | 防御性编程基本原则 | ✅ 正确且必要 |

### 1.4 CTO 指令的补充建议

CTO 指令中未提及但基于联网查证应**强烈建议**在本次一并落实的措施：

| 补充项 | 理由 | 优先级 |
|--------|------|--------|
| **启用 `mask_error_details=True`** | 只有配合此参数，TO "严禁暴露 Python 堆栈" 才能完全实现。当前仅依赖 ToolError 不够——若底层引擎抛出未预期的 `olefile.OleError` 或 `lxml.etree.LxmlError`，仍会将完整堆栈泄露给 LLM 上下文。启用后，ToolError（验证失败）透传，内部异常（引擎崩溃）脱敏为通用消息。 | ⭐⭐⭐ |
| **添加文件大小上限检查** | Firecrawl 生产指南明确要求 `MAX_FILE_SIZE = 10 * 1024 * 1024`。CTO 说"无需过度防御"，但大小上限是资源保护（防 OOM），不是安全防御。对于本地单机工具，处理一个 500MB 的 DOCX 将导致进程被杀，属于功能性崩溃。 | ⭐⭐⭐ |
| **`os.path.isfile` 替代 `os.path.exists`** | `exists` 对目录路径返回 True，`isfile` 更严格。LLM 幻觉可能传入目录路径而非文件路径。 | ⭐⭐ |
| **`os.path.expanduser` 预处理** | 处理 `~/Documents/file.docx` 格式的路径，增强 Agent 纠错宽容度。 | ⭐⭐ |

### 1.5 "无需路径遍历防御" 的边界分析

CTO 指令明确指出：

> "不需要实现复杂的沙箱隔离，也**不需要**编写繁琐的正则表达式来防御蓄意的安全攻击（如高级别路径穿越漏洞）。"

**联网查证对照**：Snyk 和 StackHawk 均将路径遍历列为 MCP 服务器的 Top 3 安全风险（参考 CVE-2025-49596, CVSS 9.4）。然而，这些风险主要针对**网络暴露的 MCP 服务**（HTTP/SSE 传输），攻击面来自远程不可信客户端。

**本项目场景**：stdio 传输、单用户、本地文件系统、仅处理用户自己的手稿文档。威胁模型是 **LLM 产生幻觉路径**（如 `D:/nonexistent/paper.docx`），而非恶意攻击者构造 `../../etc/passwd`。

**结论**：CTO 的边界决策**完全合理**。在 stdio 单机场景下，复杂的路径规范检查（`os.path.commonpath`、正则匹配）增加代码复杂度而收益接近于零。但**一个极简的 `os.path.realpath` 调用**（单行代码，零复杂度）可以作为免费午餐加入，成本极低且提供基础防御深度。

## 2. 技术细节推演

### 2.1 统一验证入口设计

**方案选择**：统一 helper 函数 `_validate_input_file()` + 每个 tool 入口显式调用。

**拒绝的方案**：装饰器模式 `@validate_input(extensions=['.docx'])`。理由：
- CTO 要求"极其轻量"，装饰器增加一层抽象间接性
- 4 个工具的参数签名不同（1-3 个参数），装饰器需要处理可变参数，增加复杂度
- 显式调用更易调试，堆栈信息更清晰

**推荐实现结构**：

```
server.py 架构变化 (after):
├── _ALLOWED_EXTENSIONS = {tool_name: ['.ext', ...]}  # 集中管理白名单
├── _MAX_FILE_SIZE = 100 * 1024 * 1024                 # 100MB 上限
├── _validate_input_file(path, allowed_exts) -> None   # raise ToolError on failure
├── convert_md_to_docx     → _validate_input_file(md_path, ['.md'])
├── extract_text_from_pdf  → _validate_input_file(pdf_path, ['.pdf'])
├── convert_pdf_to_images  → _validate_input_file(pdf_path, ['.pdf'])
└── convert_docx_to_md     → _validate_input_file(docx_path, ['.docx'])
```

### 2.2 `_validate_input_file()` 详细设计

```
def _validate_input_file(file_path: str, allowed_extensions: list[str]) -> None:
    """
    在文件 I/O 前进行轻量输入防御。任何校验失败均 raise ToolError，
    提供可指导 Agent 纠错的自然语言消息。
    """
```

**校验步骤（按顺序，短路逻辑）**：

| 步骤 | 检查项 | 错误消息示例 | 复杂度 |
|------|--------|-------------|--------|
| 1 | `file_path` 非空且为字符串类型 | `"未提供文件路径，请指定一个有效的本地文件路径"` | O(1) |
| 2 | 文件存在 (`os.path.isfile` 优先，`os.path.exists` 兜底) | `"文件不存在: D:/path/to/file.docx，请检查路径是否正确"` | O(1) |
| 3 | 扩展名白名单 (`.lower()` 大小写不敏感) | `"不支持的文件类型: .txt，该工具仅接受 .docx 文件"` | O(1) |
| 4 | 文件大小上限 | `"文件过大 (150MB)，当前限制为 100MB"` | O(1) |

**设计原则**：
- 每个校验失败独立返回，不做组合错误（单次一个错误消息更易被 LLM 理解）
- 错误消息格式：`"具体问题描述，建议操作指导"`（符合 MCP Error Handling Best Practices: "Be specific. Be actionable."）
- 不包含文件系统绝对路径细节（即使未被 mask，也应保持最小信息泄露原则）

### 2.3 扩展名白名单映射

| MCP 工具 | 必需参数 | 允许扩展名 | 说明 |
|----------|---------|-----------|------|
| `convert_docx_to_md` | `docx_path` | `.docx` | 仅接收 Word 文档 |
| `convert_md_to_docx` | `md_path` | `.md`, `.markdown` | Markdown 文件 |
| `extract_text_from_pdf` | `pdf_path` | `.pdf` | 仅接收 PDF |
| `convert_pdf_to_images_fallback` | `pdf_path` | `.pdf` | 仅接收 PDF |

### 2.4 工具函数签名变化分析

**当前模式**（反模式）：
```python
try:
    ...
except Exception as e:
    return f"转换失败: {str(e)}"
```

**CTO 要求的新模式**（FastMCP 最佳实践）：
```python
_validate_input_file(path, ['.docx'])  # 先行验证，失败则 ToolError 中断
try:
    ...
except Exception as e:
    # 内部引擎异常：raise 让 FastMCP 处理（脱敏或不脱敏取决于 mask_error_details）
    raise
```

**关键语义变化**：

| 维度 | Before (当前) | After (CTO 设计) |
|------|--------------|-----------------|
| 验证失败 | `return "转换失败: ..."` → MCP 视作**成功响应** | `raise ToolError(...)` → MCP 视作**错误响应** |
| 内部异常 | `return "转换失败: {str(e)}"` → 堆栈信息作为成功内容泄露 | `raise` → 由 FastMCP 框架决定脱敏或透传 |
| LLM 感知 | `isError: false`（语义错误） | `isError: true`（协议级错误） |

> ⚠️ **注意**：移除 `except Exception as e: return f"...失败: {str(e)}"` 后，必须启用 `mask_error_details=True`。否则内部异常（如 `olefile.OleError`）的完整 trace 将原样发送给 LLM 上下文，比当前行为更糟。

### 2.5 `mask_error_details=True` 的影响范围

启用后：

| 异常来源 | LLM 看到的消息 | 说明 |
|---------|---------------|------|
| `raise ToolError("文件不存在...")` | `"文件不存在: ..."` | ToolError **永远完整透传** |
| `raise FileNotFoundError(...)` | `"Internal error"` (脱敏) | 被 mask，LLM 无法自校准 ⚠️ |
| 底层引擎 `OleError` / `LxmlError` | `"Internal error"` (脱敏) | 被 mask ✅ |
| Python 段错误 / OOM | 进程崩溃，无响应 | MCP 客户端感知超时 |

**这意味着**：启用 `mask_error_details=True` 后，必须确保所有"LLM 需要知道以自校准"的错误都通过 `ToolError` 显式抛出。仅靠 `_validate_input_file` 不够——底层引擎也可能产生可操作的错误（如 `olefile` 解析失败时的文件损坏提示）。

### 2.6 底层引擎错误的自校准友好化

当前 `docx_to_md.py` 底层函数返回错误字符串（如 `"转换失败: DOCX 文件不存在"`）。在启用 `mask_error_details=True` 后：

- `_validate_input_file` 处理 P0-1 的文件不存在/类型错误 → ToolError ✅
- 底层引擎的 `FileNotFoundError` 被 mask → LLM 看到 `"Internal error"` ❌

**解决方案**：在 tool 函数内部，不改变底层引擎的异常，而是在 server 层 `except` 中区分：

```python
try:
    ...
except FileNotFoundError as e:
    raise ToolError(f"文件未找到: {e}")
except PermissionError as e:
    raise ToolError(f"文件访问被拒绝: {e}")
except Exception:
    raise  # 其他内部异常保持脱敏
```

这并非"过度防御"，而是确保 LLM 能对可操作错误进行自校准。

## 4. 边界与约束分析

### 4.1 什么这次不做

| 不做 | 原因 | 后续计划 |
|------|------|---------|
| 复杂的路径遍历防御 (`os.path.commonpath`、正则过滤) | CTO 明确排除，stdio 单机场景威胁模型不同 | 若未来支持 HTTP 传输再评估 |
| OLE 二进制安全边界 (P0-2) | 独立风险项，需单独设计任务 (D3) | 见风险评估 P0-2 |
| PDF 页数/图片数量上限 | 与 P1-1 (资源限制) 耦合，需整体设计资源管控 | 见风险评估 P1-1 |
| 可选输出参数 (`docx_path`, `output_dir`) 的校验 | 输出路径不需要存在性校验；父目录权限校验超出本次范围 | 低优先级 |

### 4.2 风险自评

| 风险 | 可能性 | 缓解措施 |
|------|-------|---------|
| **ToolError 改变现有 MCP 客户端行为** | 低 | Roo Code 和 Claude 等主流客户端均正确识别 `isError: true` 响应并传递给 LLM 上下文字段，与当前 `return "失败: ..."` 的行为一致 |
| **mask_error_details 导致 LLM 无法自校准未知内部错误** | 中 | 通过 server 层显式 catch `FileNotFoundError` / `PermissionError` 并转为 ToolError，确保高频可操作错误不被 mask |
| **文件大小上限误拦合法大文件** | 极低 | 100MB 上限（DOCX 场景远高于常规学术论文大小）；若需处理超大文件，可通过修改 `_MAX_FILE_SIZE` 常量调整 |
