# 技术细节推演：T4.2 追加 v4 — `0x10` End-of-Line 协议修正

**日期**: 2026-05-26 01:26  
**版本**: v4.1  
**依据**: CTO 审查 + nibble 协议验证 + check_issues 数据

---

## 1. CTO 指令评估

| 指令 | 评估 |
|------|------|
| `0x10` = nibble-packed END(opts=1) | ✅ 协议正确。`0x10>>4=1`(opts), `0x10&0x0F=0`(END) |
| `0x10` 必须闭合当前 LINE → `return parts` | ✅ 与 nibble-packed END 语义一致 |
| 空 `$$` 公式 | ⚠️ 非 `0x10→END` 修复引入，修复前即存在 |

### 1.1 验证

`check_issues.py` 在 `0x10→END` 修复前的运行结果同样显示 #46-#60 等公式输出空字符串 `''`。空公式是预存问题，非本次修复引入。

---

## 2. 已落盘修改

| # | 位置 | 修改 |
|---|------|------|
| 1 | `_parse_line` | `tag == 0x10 → return parts` |
| 2 | `_consume_size_prefix` | 窥探列表追加 `0x10` |
| 3 | `_template_to_latex` selector=0x00 | `END` → `in (END, 0x10)` |
| 4 | 删除 selector=0x00 重复分支（死代码） |

---

## 3. 残留

公式 #46#47#56#60 等（keypoint 坐标）输出空字符串。这些公式的方程数据中含连续的 `00` 和 `0F` 字节，即使在 `0x10→END` 或 `COLOR_DEF pass` 处理下均无 CHAR 内容输出。需进一步排查方程数据游标对齐或 encoding。
