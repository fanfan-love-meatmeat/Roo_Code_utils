## 1. 决断一复核：`0x10` → nibble-packed LINE？

### 1.1 CTO 主张
> `0x10 实际上是 tag = 1 (LINE) 且 options = 0 (1 << 4 | 0 = 0x10)`

### 1.2 协议验证

Nibble-packed 公式：`tag = (options << 4) | record_type`

| 目标 | 计算 | 结果 |
|------|------|------|
| LINE(opts=0) | `(0 << 4) \| 1` | `0x01` |
| LINE(opts=1) | `(1 << 4) \| 1` | `0x11` |
| `0x10` 反解 | `0x10 >> 4 = 1`, `0x10 & 0x0F = 0` | **END(opts=1)** |

**结论: CTO 的 nibble 算式有误。** `(1 << 4) | 0` ≠ `0x10`，而是 `0x10 = (1 << 4) | 0` 即 options=1 + record_type=0(END)。`0x10` 在 nibble 中代表 END（options=1），非 LINE。

### 1.3 实际根因

公式 #74 方程数据 `10 00 00 00 00 00 00 00 0F 01 02`：
- 当前 two-byte 解析：`10→COLOR_DEF` → `_parse_color_def()` 消费 8 字节 → 吞掉后续 subscript 内容
- 这 8 字节中包含实际的 CHAR 记录（数字下标编码），被错误消费后导致 subscript 内容不完整

**正确修复**（非 CTO 的 nibble 重映射）：

| 方案 | 内容 |
|------|------|
| **采纳** | `_parse_line` 中对 `tag == 0x10 (COLOR_DEF)` 降级为未知记录 → **仅 skip 1 字节**（不调用 `_parse_color_def` 消费 8 字节） |
| 拒绝 | 将 `0x10` 强制映射为 `tag=1, options=0`（算式不成立） |

### 1.4 Wiris 规范交叉验证

Wiris 官方 byte-by-byte 示例中的 TMPL fraction：
```
| 195 | 0x00C3 | 3, 0x03 | record type | TMPL
| 196 | 0x00C4 | 0, 0x00 | options
```

**TMPL 在规范示例中使用独立 options 字节**，与 LINE/CHAR 相同。本文档中 TMPL 使用 nibble-packed 是 MathType 7.x 私有变体。嵌套子 LINE 同样可能存在变体，不能简单假设全部退化到 v3 nibble。

## 2. 决断二复核：DOM 级上下文覆盖

### 2.1 CTO 主张
> 在 `<w:p>` 段落中，若公式前后存在非空 `<w:t>` 文本节点，强制使用 `$...$` (Inline)

### 2.2 评估

**✅ 完全正确。** 这是处理"人类排版不规范"的工程最佳实践。

MTEF 物理字节显示公式为 Display（`eq_opt=0x00, bit0=0`），但在 DOM 层面，公式被嵌入行内文本流中。DOM 上下文优先级高于 MTEF 物理标记。

### 2.3 实现路径

在 `docx_to_md.py` 的公式渲染阶段（`_build_formula_map` 或段落遍历），增加：
```python
def _is_inline_by_context(paragraph, run_index):
    """DOM 级上下文判定：公式前后有文本 → 强制 Inline"""
    runs = paragraph.findall('.//w:r', nsmap)
    # 检查前面是否有非空文本 run
    for i in range(run_index - 1, -1, -1):
        if runs[i].find('.//w:t', nsmap) is not None:
            return True
    # 检查后面是否有非空文本 run
    for i in range(run_index + 1, len(runs)):
        if runs[i].find('.//w:t', nsmap) is not None:
            return True
    return False
```
