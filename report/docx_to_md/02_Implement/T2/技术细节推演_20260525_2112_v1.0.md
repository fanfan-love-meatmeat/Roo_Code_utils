### 需求二：参考文献双层火控

**现状分析**：

| 检查项 | 当前代码状态 | 差距 |
|---|---|---|
| **第一层：`<w:br/>` 硬转译** | ✅ 已处理 | `_parse_run_children` L393: `elif tag in ("br", "cr"): text_parts.append("\n")` |
| **第二层：正则 `[1]`, `[2]` 清洗** | ❌ 未实现 | 需要新增全局正则后处理 |

**检索验证**：

- python-docx 官方 Schema 确认 `w:br` 是 `CT_Br` 类型，作为行内换行（Shift+Enter）存在
- Zotero 文献样式中的软回车通过 `w:br` 映射为 Markdown `\n`
- 部分 Zotero 样式（如 IEEE）会在同一段落内用 `w:br` 分隔多条文献，当前转换结果类似：
  ```
  [1] Author A. Title A. Journal A. 2024.\n[2] Author B. Title B. Journal B. 2023.
  ```
  这在 Markdown 中已可阅读，但若段落内参考文献数量多且无 `\n\n` 分隔，检索 Agent 分块时可能被切断

**修复方案**：Phase 4 输出前增加一层正则清洗

```python
def _normalize_reference_lines(md_content: str) -> str:
    """
    确保每条参考文献标识 [1], [2], ... 独占新行。
    使用 lookbehind 断言检测前方非换行，强制插入 \n。
    """
    # 匹配行中出现的 [数字] 引用标识（非行首）
    return re.sub(
        r'(?<!\n)(\[\d+\])',      # 前方不是换行的 [数字]
        r'\n\1',                   # 前方插入换行
        md_content
    )
```

**影响评估**：仅处理引用标识，不误伤行内普通 `[1]` 括号编号（因它们前方通常是空格，不会被正则为独立引用）。
