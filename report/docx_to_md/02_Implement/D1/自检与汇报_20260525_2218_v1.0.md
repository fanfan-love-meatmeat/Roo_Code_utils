## 2. 架构安全自检

### 2.1 防御机制验证

| 防御层 | 状态 | 验证方式 |
|--------|------|----------|
| L1: 扩展记录优先权 | ✅ 已实装 | `if tag == FONT_DEF/ENCODING_DEF/EQN_PREFS/COLOR_DEF` 在 `tag & 0x0F` 之前 |
| L2: Null 终止符安全 | ✅ 已实装 | `_parse_font_def` / `_parse_encoding_def` 内部 `_read_null_terminated_string()` 含 `ValueError` 兜底 |
| L2: EQN_PREFS 上界截断 | ✅ 已实装 | `self.pos = min(len(self.data), self.pos + 80)` |
| L3: 精准接管 | ✅ 已实装 | `record_type = tag & 0x0F` 仅在扩展记录判断之后执行 |
| 通用: 未知 tag 兜底 | ✅ 已实装 | else 分支仅 skip 1 字节 |

### 2.2 代码扁平化

- `_skip_header` 从 3 层嵌套 `elif` 简化为 2 层（`if/elif` 扩展记录 + `else/if` 标准记录）
- 移除死代码：原 `elif record_type == FONT_DEF` 等分支因 LINE/CHAR/TMPL 在前永不可达
- 注释精简为中文工程风格，去除非必要的装饰性英文注释

### 2.3 侧面破坏风险评估

| 风险项 | 评估 |
|--------|------|
| `_parse_font_def` 移除 style 字节后 `font_map` 键值变化 | 低风险 — `font_map` 仅用于 typeface→font 回退映射，不影响方程 AST 结构 |
| `_parse_color_def` 从 4B 增至 8B 可能超出数据边界 | 已防护 — `_read_uint16_le()` 内部含 `len(self.data)` 边界检查 |
| `_skip_header` break 条件不变 | 安全 — 标准记录 `tag & 0x0F` 分类逻辑未修改，仅重新排列判断顺序 |

### 3.2 修复效果对比

**修复前**:
```
{s\langle{}m{l{}{}\langle\rangle℈{Ȃā\langle澃\rangle碃}}\rangle}  (62 chars)
```

**修复后**:
```
Ȃā\langle햄{}\rangle  (19 chars)
```

- ✅ **元数据字节不再污染输出** — `_skip_header` 正确跳过了 FONT_DEF×6 + ENCODING_DEF×2 + EQN_PREFS×1（共 179 字节）
- ⚠️ **CHAR 值仍不正确** — 方程数据区的 SIZE 表紧凑格式被当作 AST 记录解析

### 3.3 SIZE 表残留问题

修复后方程数据从 pos=191 开始，首字节 `0C`（SUB2 类型标签），后续为 SIZE 表字节序列。当前 `_parse_size_record` 仅处理单条 SIZE (tag=0x09) 记录，不支持紧凑格式 SIZE 表。这导致：
- 260/263 公式的 CHAR 值读取自 SIZE 表字节，而非实际公式字符
- 3/263 公式（不含 SIZE 表）输出相对接近正确但仍有编码偏差

**此问题为独立缺陷，与 nibble 冲突修复正交，建议下一 Sprint 处理。**
