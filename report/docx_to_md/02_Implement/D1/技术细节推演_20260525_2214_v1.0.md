### 1.2 规范对照

Wiris 官方规范明确定义了 4 个 MTEF v5 **扩展记录** (Extended Records)，其 tag 字节值为 `0x10–0x13`：

| 十进制 | Tag 字节 | 记录名 | 低4位 (= tag & 0x0F) | 与标准记录冲突 |
|--------|----------|--------|----------------------|----------------|
| 16 | `0x10` | COLOR_DEF | `0x00` | END |
| 17 | `0x11` | FONT_DEF | `0x01` | LINE |
| 18 | `0x12` | EQN_PREFS | `0x02` | CHAR |
| 19 | `0x13` | ENCODING_DEF | `0x03` | TMPL |

**结论**: 扩展记录的低 4 位与标准 AST 记录完全同值，这是 MTEF v5 协议设计层面的固有冲突，不是编码错误。任何仅使用 `tag & 0x0F` 的分类器都无法区分扩展记录与标准记录。

### 1.3 各扩展记录格式（依据 Wiris 官方 byte-by-byte 示例）

**FONT_DEF (0x11):**
```
tag(1) | encoding_index(1) | font_name(null-terminated)
```
示例（官方文档 0x004B 处）：
```
17,0x11 | record type = FONT_DEF
4, 0x04 | index of encoding MTExtra
77,0x4D | font name "MT Extra\0"
```
⚠️ **与当前代码的差异**: 当前 `_parse_font_def()` 在 encoding_index 之后多读取了一个 `style` 字节。根据规范，encoding_index 之后直接就是 font_name。此差异会导致多字节字体名（如 GBK 编码的中文名）的首字节被误读为 style。

**ENCODING_DEF (0x13):**
```
tag(1) | encoding_name(null-terminated)
```
预定义编码：MTCode(1), Unknown(2), Symbol(3), MTExtra(4)。

**EQN_PREFS (0x12):**
```
tag(1) | options(1) | size_array_count(1) | nibble_array(variable) | style_defs(variable)
```
⚠️ **CTO 指令与规范的差异**: CTO 指令称 EQN_PREFS "理论上是 80 字节定长"，但 Wiris 官方示例明确显示其格式为 `count(1) + nibble_array(count个) + style_defs(count个)`，是**可变长**结构。80 字节是 MathType 7 默认配置的经验值，非协议保证。对于第三方工具生成或修改过的文档，该假设可能失效。

**COLOR_DEF (0x10):**
```
tag(1) | R_uint16(2) | G_uint16(2) | B_uint16(2) | spare_uint16(2)
```
共 8 字节（含 tag）。当前代码仅 skip 4 字节，缺少 2 个 uint16。

### 1.4 CTO 指令合理性评估

| CTO 指令项 | 与规范一致性 | 评估 |
|------------|-------------|------|
| 第一道防线：扩展记录绝对优先权 | ✅ 完全一致 | 正确且必要 |
| 第二道防线：Null 终止符 try-except | ✅ 完全一致 | 防御性编程最佳实践 |
| 第二道防线：EQN_PREFS 80 字节定长 | ⚠️ 部分偏差 | 80 字节是经验值，非协议保证；规范定义为可变长 |
| 第三道防线：精准接管 | ✅ 完全一致 | 先判扩展再判标准 |
| 要求回归通过率 100% | ✅ 合理目标 | nibble 修复后至少消除乱码根因 |

**综合评定**: CTO 指令的架构方向完全正确，三层防御体系设计合理。仅在 EQN_PREFS 长度处理上存在与官方规范的细微偏差，属于可接受的经验假设，不影响本次修复的正确性。

### 2.3 修改详情

#### 修改 A：`_skip_header()` 重构

**变更前逻辑:**
```
读 tag → record_type = tag & 0x0F → 命中 LINE/CHAR/TMPL 则 break
                                    → 命中 FONT_DEF/ENCODING_DEF 则跳过
```
**断裂点**: `tag=0x13 → record_type=0x03=TMPL → 误 break`。FONT_DEF/ENCODING_DEF 的 elif 分支因 LINE/CHAR/TMPL 在前，**永远不可达（死代码）**。

**变更后逻辑:**
```
读 tag → 先判 full tag 是否 ∈ {0x10, 0x11, 0x12, 0x13}
              → 是 → 走扩展记录跳过逻辑（按规范格式逐字节消费）
              → 否 → record_type = tag & 0x0F
                       → 命中 LINE/CHAR/TMPL/... → break（方程数据开始）
                       → 命中 END → break
                       → 其他 → skip 1 byte
```

#### 修改 B：`_parse_font_def()` 按规范修正

**变更前:**
```python
idx = self._read_byte()       # encoding_index
style = self._read_byte()     # ← 规范中不存在，会吃掉多字节字体名首字节
name = self._read_null_terminated_string()
```

**变更后:**
```python
idx = self._read_byte()       # encoding_index
name = self._read_null_terminated_string()  # 直接读字体名
```
移除 `style` 变量及对应读取。`_skip_header` 中的 FONT_DEF 调用保持向后兼容。

#### 修改 C：`_parse_color_def()` 补齐字节

```python
# 变更前: 仅读 4 字节
self._read_byte() * 4

# 变更后: 补齐到 8 字节 (R,G,B 各 uint16 + spare uint16)
self._read_uint16_le()  # R
self._read_uint16_le()  # G  
self._read_uint16_le()  # B
self._read_uint16_le()  # spare
```

## 3. 逻辑流转

### 3.1 修复前流程（Bug 状态）

```
OLE stream "Equation Native" raw[28:]
  │
  ├─ _skip_header()
  │    pos=0..11: 正确跳过 version + app_key + eq_options
  │    pos=12: tag=0x13 (ENCODING_DEF)
  │    record_type = 0x13 & 0x0F = 3 = TMPL
  │    TMPL ∈ break_list → BREAK ❌  ← 提前终止
  │    _parse_line() 从 pos=12 开始
  │      → "WinAllBasicCodePages\0" 被当作方程 AST
  │      → 输出乱码: {s\langle{}m{l{}{}\langle\rangle℈{...
  │
  └─ 结果: 263 公式全部输出乱码
```

### 3.2 修复后流程（预期行为）

```
OLE stream "Equation Native" raw[28:]
  │
  ├─ _skip_header()
  │    pos=0..11: 正确跳过 version + app_key + eq_options
  │    pos=12: tag=0x13 → full_tag ∈ {0x10..0x13} → 进入扩展记录处理
  │      → ENCODING_DEF → 读 null-terminated name → pos 前进
  │    pos=54: tag=0x11 → FONT_DEF → 读 encoding_index + name → pos 前进
  │    pos=110: tag=0x12 → EQN_PREFS → skip 80B → pos 前进
  │    pos=191: tag=0x0C → full_tag ∉ {0x10..0x13} → 低4位=12→SUB2
  │    SUB2 ∈ break_list → BREAK ✅  ← 正确终止在方程数据边界
  │
  ├─ _parse_line() 从 pos=191 开始
  │    → 解析 SIZE 表 + LINE/CHAR/TMPL AST
  │    → 输出正确 LaTeX（或至少不再包含元数据字节）
  │
  └─ 结果: 263 公式解析指针正确对齐
```

### 3.3 SIZE 表残留问题预警

修复后方程数据起始字节为 `0C`（SUB2 类型标签），后续字节序列为 SIZE 表紧凑格式。当前 `_parse_size_record()` 仅处理单条 SIZE (tag=0x09) 记录，该区域可能触发未知行为。**建议**: 修复后运行全量回归测试，如 SIZE 表解析仍有畸变，提独立 Issue 进下一 Sprint。

## 4. 防御机制

| 防御层 | 实现位置 | 机制 |
|--------|----------|------|
| L1: 扩展记录优先 | `_skip_header` while 循环 | `if tag in (0x10, 0x11, 0x12, 0x13)` 在 `record_type = tag & 0x0F` 之前判断 |
| L2: Null 终止符安全 | `_parse_font_def`, `_parse_encoding_def` | `try: data.index(0, pos); except ValueError: pos = len(data)` |
| L2: 动态上界截断 | `_parse_eqn_prefs` | `pos = min(len(data), pos + 80)` — 保留 80B 经验值，但增加动态上限保护 |
| L3: 精准接管 | `_skip_header` break 分支 | 仅在确认 `tag ∉ {扩展记录}` 后执行 `record_type = tag & 0x0F` 比较 |
| 通用: 未知 tag 兜底 | `_skip_header` else 分支 | 无法识别的 tag 仅 skip 1 byte，不会死循环 |
