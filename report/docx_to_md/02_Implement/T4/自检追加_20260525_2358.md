## 뢄/햄 公式 Hex Dump（根本性设计缺陷发现）

### Formula #32 (`뢄`, 227B, 俯仰角)

**方程数据**（pos 217, 10B）:
```
00D9: 01 00 02 04 84 B8 03 71 00 00
```

### Formula #33 (`햄`, 227B, 偏航角)

**方程数据**（pos 217, 10B）:
```
00D9: 01 00 02 04 84 D5 03 66 00 00
```

### 根本性解析错误

当前解析器将 `02 04 84 B8` 解释为：
```
tag=02 → options = 02>>4 = 0  ❌
typeface = 04
char_val = 0xB884 (uint16) → chr → '뢄'
```

**正确的 MTEF 格式**（与 Wiris 官方 byte-by-byte 示例一致）：
```
02    = CHAR 标签 (record_type)
04    = 独立 options 字节 (CHAR_ENC_CHAR_8)
84    = typeface (≥128 → MTCode 单字节)
B8    = MTCode 单字节值
```

Wiris 官方示例证实此格式：
```
| 202 | 0x00CA | 2, 0x02 | record type | CHAR
| 203 | 0x00CB | 4, 0x04 | options: mtefOPT_CHAR_ENC_CHAR_8
| 204 | 0x00CC | 134, 0x86 | typeface: 134-128=6
| 205 | 0x00CD | 18, 0x12 | MTCode value
```

**结论**: MTEF v5 协议中，**options 是独立字节**，紧跟在 tag 字节之后。当前全解析器使用 `tag >> 4` 提取 options 是全局性设计错误，影响所有记录类型（LINE/CHAR/TMPL/SIZE 等）。

### 修复方向

需全局修改 `_parse_line`、`_skip_header` 等所有读取 tag 的位置：
- tag 字节的低 4 位为 record_type
- tag 字节的下一字节为 options
- 不再使用 `options = tag >> 4`
