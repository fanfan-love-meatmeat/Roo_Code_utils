## 1. 深度审查验证：四项漏洞逐一核实

### 🔴 漏洞 1：Pillow WMF/EMF 渲染 — 确认存在，原方案过于乐观

**检索验证结果：**

- Pillow 官方文档明确声明：**仅在 Windows 上可读取 WMF/EMF 文件**，在 Linux/macOS 上完全不支持（见 [Pillow Docs - WMF/EMF](https://pillow.readthedocs.io/en/stable/handbook/image-file-formats.html#wmf-emf)）
- Pillow GitHub Issue [#6980](https://github.com/python-pillow/Pillow/issues/6980)：EMF 文件报 `OSError: cannot render metafile` — 即使在 Windows 上，`PIL.WmfImagePlugin` 也仅调用 GDI `drawwmf`，部分 WMF/EMF 仍会渲染失败
- Stack Overflow [Convert EMF/WMF files to PNG/JPG](https://stackoverflow.com/questions/14103891/convert-emf-wmf-files-to-png-jpg)：7 个答案一致确认 Pillow 无法独立完成 WMF→PNG 转换
- 替代方案：Inkscape (`inkscape --export-type=png`)、ImageMagick (`wand`)、Aspose (商业)

**修正方案：**

| 策略 | 优先级 | 具体做法 |
|---|---|---|
| **A. Windows GDI (pywin32)** | 首选（Windows 环境） | `win32com.client` 或 `ctypes` 调用 GDI PlayEnhMetaFile 渲染 WMF/EMF → BMP → Pillow 转 PNG。精度高，无外部依赖 |
| **B. 外部工具降级** | 备选 | 检测系统已装 `inkscape` 或 `magick`，有则调用；无则输出原始 WMF 并在 MD 中标注 `[需要手动处理: xxx.wmf]` |

**最终决策：** 由于 MCP 服务运行在 Windows 环境（`C:/Users/98514/.conda/envs/roo_mcp/python.exe`），采用**方案 A + B 降级兜底**。WMF/EMF 渲染失败时保留原文件，不阻塞整体转换流程。

### 🔴 漏洞 2：图片 rId 映射断链 — 确认缺失关键桥梁

**检索验证结果：**

- DOCX ZIP 结构认知修正：
  ```
  docx.zip
  ├── word/
  │   ├── document.xml          ← <a:blip r:embed="rId5"/>  只含 rId，不含文件名
  │   ├── _rels/
  │   │   └── document.xml.rels ← <Relationship Id="rId5" Target="media/image1.wmf"/>  桥梁
  │   └── media/
  │       ├── image1.wmf         ← 物理文件，无 rId 信息
  │       └── image2.png
  └── ...
  ```
- python-docx 暴露了此能力（[Stack Overflow](https://stackoverflow.com/questions/36235031/where-is-the-word-rels-document-xml-rels-in-python-docx-object)）：
  ```python
  doc = Document(path)
  for rId, rel in doc.part.rels.items():
      if "image" in rel.reltype:
          print(f"{rId} -> {rel.target_part.partname}")  # rId5 -> /word/media/image1.wmf
  ```

**修正方案：** 在图片提取阶段增加强制步骤 — **先解析 `word/_rels/document.xml.rels` 构建 `{rId: 物理文件名}` 映射表**，再据此从 `word/media/` 提取对应文件并重命名。同时，为每个图片维护 `{rId: disk_relative_path}` 映射，供文档遍历时引用。

### 🔴 漏洞 3：python-docx 与 lxml 双轨数据防脱序 — 确认存在严重错位风险

**检索验证结果：**

- python-docx `paragraph.text` 只返回 `w:r/w:t` 元素的文本拼接，**完全丢弃** `m:oMath`（公式）、`w:drawing`（图片）、`w:object`（OLE 对象）的内容（[python-docx 官方文档](https://python-docx.readthedocs.io/en/latest/api/text.html)）
- 若用 python-docx 取文字，再用 lxml 独立扫描 XML 取公式，二者的顺序信息完全丢失 —— 无法判断公式在段落的前、中、后位置
- python-docx 1.0+ 提供 `paragraph.iter_inner_content()` 可遍历 run 和 hyperlink，但**仍不支持 OMML 和 drawing**

**修正方案：** 放弃双轨并行。改为**直接操作底层 XML 元素 `paragraph._p`**，按 `iterchildren()` 顺序遍历所有子节点：

```
paragraph._p 的子节点顺序示例:
  w:r        → "根据公式 "
  m:oMath    → OMML 公式块 → 转 LaTeX $$...$$
  w:r        → " 可得 "
  w:drawing  → 图片引用 → rId → 查映射 → ![](images/xxx.png)
  w:r        → " 如下所示"
  w:object   → OLE 对象 (MathType) → 查映射 → ![](images/eq_001.png)
```

**实现路径：**
```python
from docx.oxml.ns import qn

def parse_paragraph_children(para, image_map):
    """按顺序遍历 paragraph._p 子节点，产出 Markdown 片段"""
    for child in para._p.iterchildren():
        tag = child.tag.split('}')[-1] if '}' in child.tag else child.tag
        
        if tag == 'r':                        # 文字块
            yield extract_run_text(child)
        elif tag == 'oMath' or tag == 'oMathPara':  # OMML 公式
            yield omml_to_latex(child)
        elif tag == 'drawing':                # 图片/公式图
            rId = extract_blip_rid(child)
            yield f'![]({image_map[rId]})'
        elif tag == 'object':                 # OLE (MathType)
            rId = extract_ole_rid(child)
            yield f'![公式]({image_map[rId]})'
```

### 🔴 漏洞 4：手写 OMML→LaTeX 的造轮子风险 — 确认，改用 XSLT 方案

**检索验证结果：**

- Microsoft 官方随 Office 分发 `OMML2MML.XSL`（路径：`C:\Program Files\Microsoft Office\root\Office16\OMML2MML.XSL`）
- [TEI Consortium](https://tei-c.org/Vault/Stylesheets/) 公开发布了 `omml2mml.xsl`（BSD/Creative Commons 双许可），经历年维护已非常成熟
- GitHub 开源项目 [plurimath/omml2mathml](https://github.com/plurimath/omml2mathml) 封装了此 XSLT 的 Ruby 版本
- MathML → LaTeX 有成熟方案：`mmltex.xsl`（来自 [mathconverter](https://github.com/oerpub/mathconverter)）或 `py_mathml_to_latex` 库

**修正方案：两步 XSLT 管道**

```
OMML (m:oMath XML)
  │
  ├─[XSLT 1] omml2mml.xsl (TEI/Microsoft, BSD 许可)
  │
  ▼
MathML
  │
  ├─[XSLT 2] mmltex.xsl (mathconverter, MIT 许可)
  │
  ▼
LaTeX 字符串 → $$...$$
```

**优势：**
- OMML→MathML 覆盖 Office 公式全集（分数、根号、矩阵、积分、上下标、括号、重音等），由 Microsoft 官方维护
- MathML→LaTeX 覆盖标准集合，由社区多年打磨
- 两个 XSLT 文件各约 20-50KB，直接嵌入项目 `tools/xslt/` 目录，零外部服务依赖
- 约 40 行胶水代码完成整个管道，替代上千行手写递归遍历

**XSLT 文件来源（均可离线内置）：**
1. `omml2mml.xsl` — 从 TEI Stylesheets 提取（BSD 许可），或从本机 Office 安装目录复制
2. `mmltex.xsl` — 从 [mathconverter](https://github.com/oerpub/mathconverter/blob/master/mathconverter/xsl_yarosh/mmltex.xsl) 提取（MIT 许可）

## 2. 文字转 Markdown 是否有问题？

**结论：没有问题。** python-docx 对段落文本（`w:r/w:t`）、粗体/斜体（`w:rPr/w:b`, `w:i`）、表格（`w:tbl`）、有序/无序列表（`w:pPr/w:numPr`）的提取均成熟可靠。Markdown pipe table、`**粗体**`、`*斜体*`、`- 列表` 等语法转换均为标准映射。

前提是采用上述"单轨 XML 遍历"方案，确保**公式、图片、文字在 Markdown 中的顺序与 Word 文档可视化顺序严格一致**。

### 3.2 核心数据流（修正后）

```
convert_docx_to_md(docx_path, output_md_path=None, image_dir=None) → str

Phase 1: 构建关系映射
│
├── 1a. zipfile 打开 DOCX
├── 1b. 解析 word/_rels/document.xml.rels → rel_map: {rId: target_path}
├── 1c. 解压 word/media/* 到 image_dir，同时构建 image_map: {rId: 磁盘相对路径}
│       对每个图片文件：
│         - PNG/JPG → 直接复制
│         - WMF/EMF → Phase 1d WMF渲染管道
│
├── 1d. [WMF 渲染] 对每个 WMF/EMF 文件：
│         try: pywin32 GDI → BMP → Pillow → PNG
│         except: 尝试 inkscape → PNG
│         except: 保留原始 WMF → 记录 _unconverted.log

Phase 2: 结构化遍历
│
├── 2a. python-docx 加载文档
├── 2b. 遍历 document.element.body 子节点（body 级：段落 + 表格）
│       对每个段落 paragraph._p.iterchildren()：
│         判断子节点标签类型：
│           w:r         → 提取文本 + 格式 (粗体/斜体)
│           m:oMath     → OMML → (XSLT管道) → LaTeX → $$...$$
│           m:oMathPara → 同上（display 公式）
│           w:drawing   → 提取 r:embed → image_map → ![](path)
│           w:object    → 提取 r:id → image_map → ![公式](path)
│       对每个表格：
│         遍历行/列 → Markdown pipe table

Phase 3: 拼接输出
│
├── 3a. 按顺序拼接 Markdown 片段
├── 3b. 写入 output_md_path (默认 DOCX 同级同名 .md)
└── 3c. 返回结果摘要
```

### 3.3 MCP Tool 接口

```python
@mcp.tool()
def convert_docx_to_md(docx_path: str, output_md_path: str = None, image_dir: str = None) -> str:
    """
    将 DOCX 文件转换为 Markdown 格式。
    - MathType/OLE 公式：导出原始 WMF/EMF 渲染图，尝试转为 PNG，Markdown 中以 ![](path) 引用
    - OMML 原生公式：通过 XSLT 管道转换为 LaTeX $$...$$ 格式
    - 普通图片：导出到图片目录并引用
    - 表格：转换为 Markdown pipe table
    
    参数:
        docx_path:       必需，DOCX 文件的绝对路径。
        output_md_path:  可选，输出 MD 文件路径。默认在 DOCX 同级目录生成同名 .md。
        image_dir:       可选，图片导出目录。默认在 DOCX 同级目录创建 {文件名}_images/ 文件夹。
    """
```

## 5. 防御机制（补充修正）

| 位置 | 机制 |
|---|---|
| `rel_map` 构建 | 检查 `rels` 中是否存在 `NULL` 目标路径（已知 Word bug），输出 WARNING |
| WMF 渲染 | 三层降级: GDI → Inkscape → 保留原文件 | 任何一层失败都不中断流程 |
| XSLT 转换 | `try/except` 包裹，失败时降级为 `[OMML公式: 转换失败]` 占位 |
| 子节点遍历 | `tag.split('}')[-1]` 处理命名空间前缀，确保跨 Office 版本兼容 |
| 图片命名冲突 | `{原文件名}_{计数器}.{ext}` 确保唯一性 |
| 递归深度 | XSLT 引擎自动处理，无需手动限制 |

## 6. 已知限制（无变化）

1. **MathType 公式不可编辑**：导出为 PNG 图片，无法在 Markdown 中二次编辑数学内容
2. **OMML → LaTeX 覆盖率**：XSLT 覆盖 Office 标准公式集合，极端复杂的自定义矩阵/重音可能存在极少数转换不佳的情况
3. **不处理 .doc 格式**：仅支持 `.docx` (Office Open XML)
4. **OLE 嵌套**：OLE 中嵌套 OLE 的场景不做递归处理
