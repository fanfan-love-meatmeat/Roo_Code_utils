# 技术细节推演：DOCX → Markdown 单向转换 MCP 工具

## 1. 检索与论证

### 1.1 DOCX 中公式的三种存在形态

| 类型 | 底层存储 | 可编程提取难度 | 本方案策略 |
|---|---|---|---|
| **OMML 公式** (Word 原生公式, `m:oMath`) | `word/document.xml` 中的 XML 元素 | 中 — 可递归遍历 OMML 树转为 LaTeX | **全文转换**：lxml 提取 → 递归转 LaTeX → `$$...$$` |
| **MathType 公式** (OLE 对象, 用户主要使用) | `word/embeddings/oleObjectN.bin` (二进制 MTEF 数据) + `word/media/` (WMF/EMF 渲染图) | 极高/不可行 — 纯 Python 无法解析 MTEF 二进制 | **降级为图片**：提取 WMF/EMF → PNG 转换 → `![eq](path)` 引用 |
| **EQ 域公式** (Word 旧格式, `EQ \f(1,2)`) | `word/document.xml` 中 `w:instrText` 字段 | 中 — 可正则提取 | **占位标记**：提取域代码原文 → `[EQ: ...]` 标记 |

### 1.2 检索结论：MathType 纯 Python 提取方程式的现实性

经 Tavily 多轮检索，**结论明确**：

- MathType 公式在 DOCX 中以 OLE 二进制对象存储，内部使用私有 MTEF (MathType Equation Format) 协议
- MathType SDK (商业闭源) 可解析 MTEF，但仅支持从 WMF 文件态转换，不支持直接解析 OLE 二进制流
- Aspose.Words (商业) 官方确认：**无法将 MathType 公式转换为 OfficeMath**
- docxlatex (开源) 仅支持 OMML 公式，不支持 MathType
- **唯一免费路径**：Word COM 自动化 (需 Windows + 已装 Word + MathType 插件)，不适合 MCP 服务端场景
- Mathpix API (商业云服务) 可通过 OCR 将公式图片转为 LaTeX，但需 API Key 和网络调用

**决策：MathType 公式 → 降级提取为 PNG 图片，在 Markdown 中以 `![equation](./images/eq_001.png)` 形式引用。**

### 1.3 图片提取方案对比

| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **zipfile 直接解压** | DOCX 是 ZIP，`word/media/` 下即原始图片 | 零依赖、最快、保真 | 需处理命名冲突 |
| python-docx `inline_shapes` | 通过关系 ID 找图片 blob | 能关联图片与文档位置 | API 不完善，获取 blob 需 hack |
| `docx2txt.process(path, img_dir)` | 第三方库，一行搞定 | 极简 | 新增依赖，不区分公式图/普通图 |

**决策：采用 `zipfile` 直接解压 `word/media/`，零额外依赖，性能最优。**

### 1.4 WMF/EMF → PNG 转换方案

MathType 的渲染图为 WMF/EMF 矢量格式。需要在无 Office 环境下转为 PNG：

- **Pillow (PIL)**：不原生支持 WMF/EMF
- **ImageMagick (Wand 绑定)**：需安装 ImageMagick 本体，过重
- **LibreOffice `soffice --headless`**：可将 WMF 转 PNG，但依赖庞大
- **`wmf2svg` + `cairosvg`**：WMF → SVG → PNG，纯 Python 路线，可行但链路长

**决策：优先尝试 Pillow + `PIL.WmfImagePlugin` 读取 WMF 并转 PNG。若失败，输出原始 WMF 文件并在 Markdown 中记录提醒。同时为用户提供 `--use-mathpix` 可选参数调用 API 做 OCR 升级。**

## 2. 架构设计

### 2.1 模块清单

```
Roo_Code_utils/tools/
├── docx_to_md.py          ← [新建] 主转换管道
│
Roo_Code_utils/server.py    ← [修改] 新增 MCP tool: convert_docx_to_md()
```

### 2.2 核心函数设计

```
docx_to_md.py:

convert_docx_to_md(docx_path, output_md_path=None, image_dir=None, use_mathpix=False) -> str
│
├── extract_images(docx_path, image_dir) -> {img_name: disk_path}
│   ├── zipfile 读取 word/media/*.wmf, *.emf, *.png, *.jpg
│   ├── 分类：MathType 相关 WMF/EMF → 尝试转 PNG
│   └── 返回 {原始名: 绝对路径} 映射
│
├── extract_text_and_structure(docx_path, image_map) -> list[Block]
│   ├── python-docx 读取段落/表格/列表
│   ├── lxml 并行解析 word/document.xml 检测公式
│   ├── 遇到 m:oMath → 调用 omml_to_latex()
│   ├── 遇到 OLE 对象引用 → 查 image_map 映射为 ![eq](path)
│   └── 返回有序 Block 列表
│
├── omml_to_latex(omml_element) -> str
│   ├── 递归遍历 OMML XML 树
│   ├── m:f → \frac{}{},  m:rad → \sqrt{},  m:sSup → ^{}
│   └── 返回 LaTeX 字符串
│
└── blocks_to_markdown(blocks, image_dir) -> str
    ├── 段落 → 空行分隔
    ├── 标题 → #/##/### 前缀
    ├── 表格 → Markdown pipe table
    ├── 公式 → $$...$$ / $...$
    └── 图片 → ![...](images/...)
```

### 2.3 MCP Tool 接口定义

```python
# server.py 新增
@mcp.tool()
def convert_docx_to_md(docx_path: str, output_md_path: str = None, image_dir: str = None) -> str:
    """
    将 DOCX 文件转换为 Markdown 格式。
    - MathType 公式：导出为 PNG 图片，保存到指定图片目录，Markdown 中以 ![](path) 引用
    - OMML 原生公式：转换为 LaTeX $$...$$ 格式
    - 普通图片：导出到图片目录并引用
    
    参数:
        docx_path:       必需，DOCX 文件的绝对路径。
        output_md_path:  可选，输出 MD 文件路径。默认在 DOCX 同级目录生成同名 .md。
        image_dir:       可选，图片导出目录。默认在 DOCX 同级目录创建 {文件名}_images/ 文件夹。
    """
```

### 2.4 逻辑流转

```
用户调用 convert_docx_to_md(docx_path, image_dir="D:/output/imgs")
│
├─ Step 1: zipfile 打开 DOCX，读取 word/document.xml 到内存
├─ Step 2: zipfile 提取 word/media/* 到 image_dir
│          ├── MathType WMF/EMF → pillow 尝试转 PNG
│          └── 构建 image_map: {rId: "eq_001.png"}
├─ Step 3: python-docx 遍历 body 元素
│          ├── 段落 → 检测内部 run 是否含公式
│          │   ├── 含 m:oMath → omml_to_latex() → $latex$
│          │   ├── 含 w:object (OLE) → 查 image_map → ![eq](path)
│          │   └── 纯文本 → 直接输出 + 粗体/斜体标记
│          ├── 表格 → Markdown pipe table
│          └── 图片 → 查 image_map → ![img](path)
├─ Step 4: 拼接 Markdown，写入 output_md_path
└─ 返回: "转换成功！Markdown: xxx.md, 图片: N 张 → D:/output/imgs/"
```

## 3. 防御机制

| 位置 | 机制 |
|---|---|
| `extract_images()` | 检查 `word/media/` 是否存在；图片命名冲突时自动追加 `_N` 后缀 |
| `omml_to_latex()` | 递归深度限制 50 层，超出时降级为 `[复杂公式]` 占位；Unknown OMML 标签记录 WARNING 日志 |
| WMF→PNG 转换 | `try/except` 包裹，失败时保留原始 WMF 并写入 `_unconverted.log` |
| `convert_docx_to_md()` | 入口 `try/except` 全局兜底，返回 `"转换失败: {detail}"` |
| 大文件保护 | 单次处理上限 50MB，超出提示用户分割文档 |
| image_dir 安全性 | 使用 `os.path.abspath` + `os.path.realpath` 防止路径穿越 |

## 6. 已知限制（透明声明）

1. **MathType 公式不可编辑**：导出为 PNG 图片，非 LaTeX，无法在 Markdown 中二次编辑数学内容
2. **复杂 WMF 可能转 PNG 失败**：依赖 Pillow 的 WMF 支持，部分矢量图可能失败
3. **嵌套 OLE 对象不支持**：OLE 中嵌套 OLE 的场景不做递归处理
4. **不处理 .doc 格式**：仅支持 `.docx` (Office Open XML)
5. **MathType 公式识别依赖文件名启发式**：无法 100% 区分普通图片和 MathType 渲染图
