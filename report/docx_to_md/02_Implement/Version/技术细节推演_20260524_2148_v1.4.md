## 1. 四项静默吞没缺陷验证

### 📊 实证数据

对该农业论文 DOCX 的 XML 结构分析结果：

| 指标 | 数量 |
|---|---|
| `w:object` (MathType OLE) | 127 |
| `w:drawing` (普通图片) | 24 |
| `m:oMath` (OMML 原生公式) | 0 |
| `mc:AlternateContent` | **0** |
| `w:ins` (修订插入) | **147** |
| `w:del` (修订删除) | **92** |
| oleObject 关系类型 | 254 |
| image 关系类型 | 141 |

---

### 🔴 缺陷 1：mc:AlternateContent 包裹标签未处理 — 部分成立

**检索结论**：Microsoft 官方文档确认 Word 会用 `<mc:AlternateContent>` 包裹 OMML 公式和 fallback 图片（[MS-ODRAWXML §3.5](https://learn.microsoft.com/en-us/openspecs/office_standards/ms-odrawxml/)）。结构为：

```xml
<mc:AlternateContent>
  <mc:Choice Requires="wpg">
    <w:drawing>...</w:drawing>  <!-- OMML 公式的 DrawingML 表示 -->
  </mc:Choice>
  <mc:Fallback>
    <w:pict>
      <v:shape ...>
        <v:imagedata r:id="rIdX"/>
      </v:shape>
    </w:pict>
  </mc:Fallback>
</mc:AlternateContent>
```

**本文件实证**：AlternateContent = 0 次。说明该文档未使用此包裹。但 **对其他来源的 DOCX 仍需防御**。

**修复方案**：在 `_parse_paragraph_children` 中新增 `AlternateContent` 标签匹配，进入时应取 `<mc:Choice>` 子节点递归处理，`<mc:Fallback>` 作为降级。

---

### 🔴 缺陷 2：修订模式包裹标签 (w:ins / w:del) — **确认存在**

**检索结论**：
- `python-docx` 无 Track Changes API（[SO 确认](https://stackoverflow.com/questions/38247251/how-to-extract-text-inserted-with-track-changes-in-python-docx)）
- `docx-revisions` 库专门处理此场景，核心逻辑：遍历时遇到 `w:ins` 则递归进入子节点，遇到 `w:del` 则跳过
- Stack Overflow 上的主流做法：`body.xpath('./w:ins//w:p')` 提取插入的段落

**本文件实证**：w:ins=147, w:del=92。虽然当前公式未受此影响（公式在原始文档中），但 **任何通过修订插入的公式/图片/文字都会被当前代码静默跳过**。

**修复方案**：在单轨遍历中增加 `w:ins` 和 `w:del` 的显式处理：
- `w:ins` → 递归进入其子节点（接受修订）
- `w:del` → 跳过（拒绝删除）

---

### 🔴 缺陷 3：XSLT 异常掩盖 — **确认存在**

**检索结论**：无需联网验证 —— 代码本身存在此逻辑。

```python
latex = omml_to_latex(child)
if latex and latex != "[公式: 转换异常]":
    parts.append(...)
# 否则：静默丢弃，不留任何痕迹
```

虽然本文件无 OMML 公式（m:oMath=0），此分支未触发，但 **防御性编程要求保留失败痕迹**。

**修复方案**：失败时输出 `[XSLT转换失败]` 或 `[公式: OMML解析异常]` 可见标记，而非完全丢弃。

---

### 🔴 缺陷 4：rId 关系类型过滤过于严格 — **确认存在**

**检索结论**：MathType/OLE 对象的关系类型为 `.../relationships/oleObject`，指向 `word/embeddings/oleObjectN.bin`（二进制 MTEF 数据），**不是** `.../relationships/image`。

当前代码：
```python
if "image" in rel.reltype:  # 只匹配 image 类型
    ...
```

**本文件实证**：oleObject 关系 254 条，image 关系 141 条。VML 渲染的 WMF 图片同时注册了 image 关系，所以当前公式能够命中。但 **纯 OLE 对象（无 WMF 渲染）会完全丢失**。

**修复方案**：扩展 `_build_image_map` 的关系类型判断，同时收录 `image` 和 `oleObject` 类型，构建 `{rId_ole → [关联的 image rIds]}` 的辅助映射。

## 2. 修复方案汇总

| # | 缺陷 | 修复指令 |
|---|---|---|
| 1 | AlternateContent 跳过 | `tag == "AlternateContent"` → 取 `<mc:Choice>` 子节点递归处理 |
| 2 | w:ins/w:del 跳过 | `tag == "ins"` → 递归进入子节点；`tag == "del"` → 跳过 |
| 3 | XSLT 失败静默 | 失败时输出 `[公式: OMML转换异常]` 标记 |
| 4 | oleObject rel 过滤 | `"image" in reltype or "oleObject" in reltype` |
