## 0. 变更动因

**v2.x 的 OCR 路线存在根本性缺陷：**

| 问题 | 说明 |
|---|---|
| PNG 模糊且比例不定 | WMF→PNG 依赖 Pillow GDI 渲染，缩放比例不可控，小公式清晰度不足 |
| OCR 精度风险 | pix2tex 对模糊小图识别率下降；复杂矩阵/上下标易出错 |
| 显存 + 推理延迟 | ~500MB VRAM，每公式 1-2s，200 公式 = 3-5 分钟 |

## 1. 技术方案调研

### 1.1 方案矩阵

| 方案 | 技术栈 | 精度 | 速度 | 依赖 | MCP 适用性 |
|---|---|---|---|---|---|
| **A. transpect/mathtype-extension** | Java + JRuby + XML Calabash | 工业级 100% | 中 | JVM 全家桶 (~200MB) | ❌ 过于重型 |
| **B. Word COM (pywin32)** | Python + Word + MathType SDK | 100% | 中 | Office + MathType 许可证 | ⚠️ 本机已就绪 (可用作降级) |
| **C. MTEF 二进制解析 (Python)** | Python `struct` + 状态机 | 高 (常见公式 >99%) | 极快 (ms/公式) | **零外部依赖** | ✅ 最佳 |
| **D. pix2tex OCR (v2.x)** | Python + PyTorch | 中 (90-95%) | 慢 (1-2s/公式) | GPU + 500MB 模型 | ⚠️ 当前保留为降级 |

### 1.2 方案 C 可行性论证

**MTEF 二进制结构概述（来自 MathType SDK 文档）：**

```
oleObjectN.bin (DOCX zip: word/embeddings/)
│
├── [0-27]  28 字节 OLE 对象头
│   └── 包含 OLE 版本、格式标识、大小
│
└── [28-]   MTEF 数据流（MathType Equation Format）
    │
    ├── MTEF Header Record
    │   ├── Version (1 byte): v3=3, v5=5
    │   ├── Platform (1 byte): Mac/Win
    │   └── Product/ProductVersion
    │
    ├── SIZING Record (可选)
    │   └── 方程渲染尺寸信息
    │
    ├── Equation Records (递归)
    │   ├── LINE / CHAR records
    │   ├── TEMPLATE records (分数、根号、矩阵)
    │   │   └── tag → 类型 (FRACTION=0x12, RADICAL=0x15, SUBSCRIPT=0x23...)
    │   └── PILE / MATRIX records
    │
    ├── FONT / ENCODING records
    │   └── 字体编码映射 (Symbol, MT Extra, etc.)
    │
    └── END Record (tag=0x00)
```

**参考实现：**

| 参考 | 语言 | 说明 |
|---|---|---|
| `danielrendall/Metaphor` | Java | 完整 MTEF 解析器，~30 commits，含单元测试 |
| `jure/mathtype` (Ruby gem) | Ruby | transpect 使用的核心解析引擎 |
| MathType SDK 文档 | C++ | Wiris 官方公开 API 文档 |

### 1.3 实现策略

**从 oleObject.bin 到 LaTeX 的三步管道：**

```
oleObjectN.bin (bytes)
  │
  ├─[Step 1: 剥离 OLE 头]
  │   取 bytes[28:] → MTEF 字节流
  │
  ├─[Step 2: MTEF 解析器]
  │   状态机逐字节读取 → AST (抽象语法树)
  │   节点类型: LINE, CHAR, TEMPLATE(FRACTION, RADICAL, SUB/SUPER, MATRIX...)
  │
  └─[Step 3: AST → LaTeX]
      递归遍历:
        FRACTION → \frac{num}{den}
        RADICAL  → \sqrt{...}
        SUPER    → ^{...}
        SUB      → _{...}
        MATRIX   → \begin{matrix}...\end{matrix}
        CHAR     → 字体编码查表 → LaTeX 符号
```

**Python 实现预估：**
- MTEF 解析器核心：~300 行（`struct` 解包 + 状态机）
- 符号查表：~200 行（希腊字母、数学符号映射）
- LaTeX 生成器：~100 行（递归遍历 AST）
- 总计：~600 行，**零外部依赖**

## 3. 架构变化

```
Phase 1: 提取 oleObject.bin (NEW)
├── zipfile 读取 word/embeddings/oleObject*.bin
├── 对照 rels 映射 → {rId_ole: ole_bin_path}
└── 构建 formula_map: {rId_ole: latex_string}

Phase 2: 文档遍历（简化）
├── w:object → rId_ole → formula_map[rId_ole] → $$latex$$
├── w:drawing → rId_img → image_map[rId_img] → ![]()
└── 不再需要 OCR_BLOCK/OCR_INLINE 占位符（直接写入 LaTeX）

Phase 3: 图片分流（保留）
└── figures/ + formulas/ 子目录

Phase 4: 输出 MD
```

**删除的模块：**
- ❌ `wmf_to_png()` — 不再需要 WMF 渲染
- ❌ `_ocr_formulas_batch()` — 不再需要 OCR 并发管线
- ❌ `_ocr_single()` / `_get_ocr_model()` — 不再需要 pix2tex 模型
- ❌ `_clean_latex()` — 不再需要 OCR 输出清洗
- ❌ OCR_BLOCK/OCR_INLINE 占位符 — 直接写入 LaTeX

**减少依赖：**
- ❌ `pix2tex` — 不再必需
- ❌ `torch` — 不再必需（保留为降级可选）
