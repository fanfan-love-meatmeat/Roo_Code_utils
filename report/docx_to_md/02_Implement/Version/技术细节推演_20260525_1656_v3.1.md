### 0.1 x86/x64 架构冲突审计

| 检查项 | 结果 |
|---|---|
| Python (roo_mcp) | **64 位** (`struct.calcsize('P')*8 = 64`) |
| Office 2024 | **64 位** (ProPlus 2024 Volume) |
| `System\MT6.dll` (根目录) | **32 位** (PE machine=0x14C) — 不可用 |
| `System\32\MT6.dll` | **32 位** — 不可用 |
| `System\64\MT6.dll` | **64 位** (PE machine=0x8664) — ✅ 可用 |

**结论**：`C:\Program Files (x86)\MathType\System\64\MT6.dll` 是 MathType 7 安装包自带的原生 64 位编译版本，64 位 Python 可直接通过 `ctypes.CDLL` 加载，无需 32 位 IPC 子进程。

### 0.2 64 位 ctypes 指针截断防御

**检索结论**：[SO #16635259](https://stackoverflow.com/questions/16635259) 等大量案例确认：在 64 位 Python 中，若不声明 `argtypes` 和 `restype`，ctypes 默认将 C 函数的**所有参数和返回值**视作 32 位 `c_int` 处理。当 `mtef_bytes` 或 `buf` 分配在高地址（>4GB）时，高位 32 位被截断，导致 MT6.dll 访问非法内存 → **进程级崩溃 (Segfault)**。

**修复**：强制声明完整函数原型。

## 1. 方案验证

### 1.1 olefile 读取 OLE 复合文档 → 确认可行

**证据**：[Stack Overflow #61979661](https://stackoverflow.com/questions/61979661) 确认 DOCX 内 `word/embeddings/oleObjectN.bin` 是 OLE Structured Storage 格式，`olefile` 可直接解析：

```python
import olefile
from zipfile import ZipFile

with ZipFile(docx_path) as zf:
    with zf.open("word/embeddings/oleObject1.bin") as f:
        ole = olefile.OleFileIO(f)
        # MathType OLE 对象 → "Equation Native" 流
        mtef_raw = ole.openstream("Equation Native").read()
        mtef_data = mtef_raw[28:]  # 跳过 28 字节 OLE 头
```

### 1.2 MT6.dll API → 确认有完整文档

**证据**：[Wiris MathType SDK 官方文档](https://docs.wiris.com/en_US/mathtype-sdk-technical-documentation/mathtype-api-documentation) 详细记录了：

| API 函数 | 用途 |
|---|---|
| `MTAPIConnect(options, timeout)` | 初始化 API（通过 `MTInitAPI` 调用） |
| `MTXFormSetTranslator(index, tdl_name)` | 设置翻译器（如 `LaTeX.tdl`） |
| `MTXFormEqn(src, srcFmt, srcData, len, dst, dstFmt, ...)` | **核心转换**：MTEF 二进制 → 文本/LaTeX |

**参数常量**（来自 SDK）：
```
src/dst: 2 = mtxfmLOCAL (本地内存操作)
srcFmt:   0 = mtxfmMTEF (MTEF 二进制格式)
dstFmt:   4 = mtxfmTEXT (文本格式，需配合 Translator)
```

### 1.3 ctypes 调用 MT6.dll → 确认架构正确

**证据**：GitHub [xuan2261/WordToLaTeX-Converter](https://github.com/xuan2261/WordToLaTeX-Converter) 使用 C# + MathType SDK 实现了同一管道，证明 `MT6.dll` 可通过外部 FFI 调用。

**Python ctypes 等价实现时需要注意的参数约束：**

根据 Wiris SDK 文档，`MTXFormEqn` 签名和调用规则：

```
status = MTXFormEqn(
    src=2,        # mtxfmLOCAL
    srcFmt=0,     # mtxfmMTEF
    srcData,      # bytes
    srcDataLen,   # len
    dst=2,        # mtxfmLOCAL
    dstFmt=4,     # mtxfmTEXT (via translator)
    dstBuffer,    # c_char_p buffer
    dstBufSize,   # buffer size
    dstPath=NULL, # not writing to file
    &actualLen    # output length
)
# returns: 0=mtOK, else error
```

**关键纠正**：原推演中的 `olefile` 用法有一个重要细节——`MT6.dll` 的函数使用 **cdecl** 调用约定（非 stdcall），因此必须用 `ctypes.CDLL` 而非 `ctypes.WinDLL`。此外，`MTAPIConnect` 的参数类型以及所有字符串参数应使用 `ctypes.c_char_p` 传递。

## 2. 终案架构

### 2.1 三步管道

```
oleObjectN.bin (DOCX zip: word/embeddings/)
  │
  ├─[Step 1: olefile 解包]  millisecond
  │   olefile.OleFileIO(bin) → openstream("Equation Native")
  │   → raw[28:] → 纯净 MTEF 字节流
  │
  ├─[Step 2: ctypes → MT6.dll 翻译]  millisecond
  │   MTAPIInit() → MTXFormSetTranslator("LaTeX.tdl")
  │   → MTXFormEqn(src=MTEF) → LaTeX 字符串
  │
  └─[Step 3: 清洗 + 写入 MD]  microsecond
       _clean_latex() → $$...$$  → Markdown
```

### 2.2 完整代码骨架

```python
import olefile
import ctypes
import threading
from zipfile import ZipFile

# 常量（来自 MathType SDK）
MTXFMLOCAL = 2
MTXFMMTEF  = 0
MTXFMTEXT  = 4

# 全局单例 + 线程安全锁
_mt6 = None
_mt6_lock = threading.Lock()


def _load_mt6():
    """全局懒加载 MT6.dll，线程安全双重检查 + argtypes 封印"""
    global _mt6
    if _mt6 is None:
        with _mt6_lock:
            if _mt6 is None:
                _mt6 = ctypes.CDLL(
                    r"C:\Program Files (x86)\MathType\System\64\MT6.dll"
                )
                # [核心] 强制声明 64 位完整原型，防止高位地址截断 → Segfault
                _mt6.MTAPIConnect.argtypes = [ctypes.c_int, ctypes.c_int]
                _mt6.MTAPIConnect.restype = ctypes.c_int

                _mt6.MTXFormSetTranslator.argtypes = [ctypes.c_int, ctypes.c_char_p]
                _mt6.MTXFormSetTranslator.restype = ctypes.c_int

                _mt6.MTXFormEqn.argtypes = [
                    ctypes.c_int,  ctypes.c_int,                       # src, srcFmt
                    ctypes.c_char_p, ctypes.c_int,                     # srcData, srcDataLen
                    ctypes.c_int,  ctypes.c_int,                       # dst, dstFmt
                    ctypes.c_char_p, ctypes.c_int,                     # dstData, dstBufSize
                    ctypes.c_char_p,                                    # dstPath (NULL)
                    ctypes.POINTER(ctypes.c_long),                     # dstDataLenOut
                ]
                _mt6.MTXFormEqn.restype = ctypes.c_int

                # 初始化连接
                _mt6.MTAPIConnect(0, 30)
                _mt6.MTXFormSetTranslator(4, b"LaTeX.tdl")
    return _mt6


def mtef_to_latex(mtef_bytes: bytes) -> str:
    """MTEF 二进制 → LaTeX（带动态扩容重试）"""
    mt6 = _load_mt6()
    buf_size = 8192

    for _ in range(3):  # 最多扩容 3 次: 8K → 16K → 32K
        buf = ctypes.create_string_buffer(buf_size)
        out_len = ctypes.c_long(0)

        status = mt6.MTXFormEqn(
            MTXFMLOCAL, MTXFMMTEF, mtef_bytes, len(mtef_bytes),
            MTXFMLOCAL, MTXFMTEXT, buf, buf_size,
            None, ctypes.byref(out_len)
        )

        if status == 0:
            return buf.value.decode("utf-8", errors="replace")
        elif out_len.value >= buf_size:
            buf_size *= 2  # 缓冲区不足 → 扩容重试
            continue
        else:
            return f"[公式: MT6 错误码 {status}]"

    return f"[公式: MT6 缓冲区溢出 (>{buf_size}B)]"
```

### 2.3 Phase 流程图

```
Phase 1: 提取 oleObject.bin + 构建 formula_map
├── zipfile 遍历 word/embeddings/oleObject*.bin
├── 对照 rels → {rId_ole: ole_bin_entry}
├── 并发调用 extract_and_convert() → formula_map: {rId_ole: latex}
└── 时间预估: 127 公式 × 1ms/个 ≈ 0.1 秒

Phase 2: 文档遍历（直接写入 LaTeX，无需占位符）
├── w:object → rId_ole → formula_map[rId] → $$latex$$
├── w:drawing → rId_img → image_map[rId] → ![]()
└── w:ins/del/AlternateContent 兼容（保留）

Phase 3: 图片分流
└── figures/ + formulas/ 子目录

Phase 4: 输出 MD
└── 写入文件
```
