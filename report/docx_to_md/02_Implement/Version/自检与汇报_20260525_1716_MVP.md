## 根因分析

### 现象

`MTAPIConnect` 返回成功 (0)，`MTXFormSetTranslator` 返回成功 (0)，`MTXFormReset` 返回成功 (0)。但 `MTAPIVersion` 返回 0x0（SDK 文档规定 MathType 7.x 应返回 0x070x）。此后所有 `MTXFormEqn` 调用一致返回 -9999。

### 推断

MathType 7.x 的 `MT6.dll` **不再是独立可调用的 SDK 核心库**。WIRIS 收购 Design Science 后，MathType 7 重构了底层架构。该 DLL 可能：

1. **强依赖 MathType 宿主进程** (`MathType.exe`)，在无窗口/进程上下文中拒绝工作
2. **仅作为 Word COM Add-in 的内部模块**编译，不暴露完整的独立 SDK 接口
3. **API 版本检测返回 0x0** 明确指示此次发布的 DLL 不支持传统 `MTXFormEqn` 内存转换管道

### 证据链

- SDK 文档说明：`MTAPIVersion` 返回 "Version is 0 if API set unknown"
- 文档警告：`MTAPIConnect` "may not be called from MathPage.WLL, use the MathPage.WLL function MTInitAPI instead"
- 文档确认：MT6.DLL "manages all communications with MathType itself, launching MathType when necessary"
- 没有任何公开案例显示 MathType 7.x 的 MT6.dll 可被独立 ctypes 调用成功

## 结论

**ctypes + MT6.dll 路径在 MathType 7.8 上不可行。** DLL 加载和 API 握手正常，但核心转换引擎在无 MathType 进程上下文下拒绝工作。

## 推荐

v3.x 方案需回退路线：

| 优先级 | 方案 | 可行性 |
|---|---|---|
| 1 | **回归 v2.3 OCR 管线** (WMF→PNG→pix2tex) | ✅ 已验证 |
| 2 | **Word COM pywin32** 自动化 (启动 Word → 获取 LaTeX) | ⚠️ 慢但 100% 准确 |
| 3 | **纯 Python MTEF 解析器** (自建/移植 Ruby mathtype gem) | ⚠️ 工程量大 |

## 附录：破解版排查

| 检查项 | 结果 |
|---|---|
| MathType.exe 数字签名 | **未签名** (NotSigned) — 破解入口 |
| MT6.dll (32/64) 数字签名 | ✅ Valid (WIRIS) — DLL 未被篡改 |
| MT6.dll FileVersion | 2024.2.21, ProductVersion 7.8.0 |
| MT6.dll IsPatched | False — 代码未修改 |
| 破解方式 | `Language\msw*.dll` (12 个 shim DLL) 注入 MathType.exe 拦截许可证检查 |
| 注册表 License 键 | 无 (Design Science/WIRIS 均无) |
| MathType.exe 预启动后测试 | **依然 APIVersion=0x0，Connect=0，Eqn=-9999** |

### 破解版结论

破解仅作用于 `MathType.exe` 进程内（通过 msw*.dll 注入拦截许可证弹窗），不影响 `MT6.dll` 的 API 行为。DLL 的 `MTXFormEqn` 在 MathType 7.x 中**需要 MathType.exe 作为宿主进程才能完整初始化**（文档明确：`"This DLL manages all communications with MathType itself, launching MathType when necessary"`）。在 ctypes 直接加载的场景下，DLL 无法启动/连接到 MathType.exe 的完整进程上下文，导致 API 核心未初始化（APIVersion=0x0）。这与是否破解无关 — 正版 MathType 7.x 的 MT6.dll 行为相同。
