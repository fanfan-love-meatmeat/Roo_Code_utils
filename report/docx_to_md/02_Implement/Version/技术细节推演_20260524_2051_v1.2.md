## 1. 第二轮审查：三项新增漏洞逐一验证

### 🔴 漏洞 5：表格单元格内嵌公式/图片的递归遍历 — 确认存在

**现实证据：**

- DOCX 的 `w:tc`（表格单元格）内部直接包含 `w:p`（段落），段落内同样可以出现 `m:oMath`、`w:drawing`、`w:object` 等任意元素（见 [python-docx #213](https://github.com/python-openxml/python-docx/issues/213)，用户展示在单元格段落中插入 OMML 的场景）
- python-docx 的 `cell.text` 仅调用 `paragraph.text` 拼接，同样会丢弃非文本元素
- 学术论文表格中嵌公式是常见场景（如实验数据表格中的统计公式、矩阵等）

**修正方案：** 表格单元格不调用 `cell.text`，改为对 `cell._tc` 下的所有 `w:p` 子节点逐个调用 `parse_paragraph_children()`，与正文段落使用同一套递归解析逻辑。

```
遍历 body 子节点:
├── w:p  → parse_paragraph_children(para, image_map)
├── w:tbl → 遍历行/列
│            └── w:tc → 遍历内部 w:p → parse_paragraph_children(cell_para, image_map)
│                       └── Markdown 表格内使用 <br> 换行, $$ 公式独立占格
└── ...
```

**Markdown 表格内公式/图片的表示策略：**
- 公式：`$$...$$` 或 `$...$` 嵌入单元格
- 图片：`![](path)` 嵌入单元格
- 多段落 → `<br>` 拼接

### 🔴 漏洞 6：OLE 对象（MathType）的 VML 命名空间陷阱 — 确认存在

**现实证据：**

- 在 python-docx GitHub Issue [#914](https://github.com/python-openxml/python-docx/issues/914) 中，真实 DOCX 内 MathType 的 OLE 结构为：

```xml
<w:object w:dxaOrig="1534" w:dyaOrig="961">
  <v:shapetype id="_x0000_t75" ...>
  <v:shape id="_x0000_i1025" type="#_x0000_t75" style="width:76.75pt;height:48.25pt" o:ole="">
    <v:imagedata r:id="rId11" o:title=""/>
  </v:shape>
  <o:OLEObject Type="Embed" ProgID="Equation.DSMT4" 
               ShapeID="_x0000_i1025" DrawAspect="Content" 
               ObjectID="_1672040852" r:id="rId12"/>
</w:object>
```

**关键发现：**
- 公式渲染图的 `r:id` 存储在 `v:imagedata/@r:id`，**不在** `w:object` 上
- `v:` 前缀对应命名空间 `urn:schemas-microsoft-com:vml`
- `o:` 前缀对应命名空间 `urn:schemas-microsoft-com:office:office`
- 不使用命名空间去 `find()` 将静默返回 `None`（[Stack Overflow 确认](https://stackoverflow.com/questions/62110439/how-to-use-python-xml-findall-to-find-vimagedata-rid-rid7-otitle-1-ren)）
- `o:OLEObject/@ProgID` 值为 `Equation.DSMT4` 可用来区分 MathType 和其他 OLE 对象（如 Excel 图表）

**修正方案：**

```python
# 在模块顶部注册所有必需命名空间
NSMAP = {
    'w':   'http://schemas.openxmlformats.org/wordprocessingml/2006/main',
    'r':   'http://schemas.openxmlformats.org/officeDocument/2006/relationships',
    'm':   'http://schemas.openxmlformats.org/officeDocument/2006/math',
    'v':   'urn:schemas-microsoft-com:vml',
    'o':   'urn:schemas-microsoft-com:office:office',
    'wp':  'http://schemas.openxmlformats.org/drawingml/2006/wordprocessingDrawing',
    'a':   'http://schemas.openxmlformats.org/drawingml/2006/main',
    'pic': 'http://schemas.openxmlformats.org/drawingml/2006/picture',
}

def extract_ole_image_rid(w_object_element):
    """从 w:object 中提取 MathType 渲染图的 rId"""
    # 方式 1：v:shape/v:imagedata
    imagedata = w_object_element.find('.//v:imagedata', NSMAP)
    if imagedata is not None:
        rid = imagedata.get('{http://schemas.openxmlformats.org/officeDocument/2006/relationships}id')
        if rid:
            return rid
    
    # 方式 2：直接查 o:OLEObject 的 r:id（备用）
    ole = w_object_element.find('o:OLEObject', NSMAP)
    if ole is not None:
        return ole.get('{http://schemas.openxmlformats.org/officeDocument/2006/relationships}id')
    return None

def is_mathtype_ole(w_object_element):
    """判断 OLE 对象是否为 MathType 公式"""
    ole = w_object_element.find('o:OLEObject', NSMAP)
    if ole is not None:
        progid = ole.get('ProgID', '')
        return progid.startswith('Equation')
    return False
```

### 🔴 漏洞 7：XSLT 文件绝对路径 vs 工作目录 — 确认存在

**现实证据：**

- MCP Server (`server.py`) 由 Kilo Code 作为子进程唤起，工作目录是**宿主项目的根目录**（如 `D:/FJL/Projects/xxx/`），不是 `Roo_Code_utils/` 目录
- 若在 `docx_to_md.py` 中使用 `"xslt/omml2mml.xsl"` 相对路径，Python 会相对于 `os.getcwd()` 解析，必然报 `FileNotFoundError`
- 标准解决方案：使用 `__file__` 锚定模块自身位置（[Python 官方最佳实践](https://docs.python.org/3/library/os.path.html)，[Stack Overflow 高票答案](https://stackoverflow.com/questions/918154/relative-paths-in-python)）

**修正方案：**

```python
# docx_to_md.py 顶部
import os

_MODULE_DIR = os.path.dirname(os.path.abspath(__file__))
_XSLT_DIR = os.path.join(_MODULE_DIR, 'xslt')

OMML2MML_XSL = os.path.join(_XSLT_DIR, 'omml2mml.xsl')
MMLTEX_XSL  = os.path.join(_XSLT_DIR, 'mmltex.xsl')

def load_xslt(path):
    """加载 XSLT 样式表，失败时抛出明确错误"""
    if not os.path.exists(path):
        raise FileNotFoundError(
            f"XSLT 样式表缺失: {path}\n"
            f"请确保 tools/xslt/ 目录包含 omml2mml.xsl 和 mmltex.xsl"
        )
    return etree.parse(path)
```

### 2.3 主数据流

```
convert_docx_to_md(docx_path, output_md_path=None, image_dir=None) → str

Phase 0: 路径锚定
├── _MODULE_DIR = os.path.dirname(os.path.abspath(__file__))
├── XSLT 文件 = os.path.join(_MODULE_DIR, 'xslt', 'omml2mml.xsl')

Phase 1: 关系映射构建
├── 1a. python-docx 加载文档 → document.part.rels
├── 1b. 遍历 rels → rel_map: {rId: target_part_name}
│       (例: "rId11" → "/word/media/image3.wmf")
├── 1c. zipfile 解压 word/media/* 到 image_dir
│       + 根据 rel_map 重命名为 {rId}.{ext} → image_map: {rId: disk_path}
│       + 对 WMF/EMF 执行 WMF 渲染管道 (GDI → Inkscape → 保留)

Phase 2: 文档树遍历 (单轨 XML)
├── document.element.body.iterchildren():
│   ├── w:p  → parse_paragraph_children(para, image_map)
│   │          按 para._p.iterchildren() 顺序遍历:
│   │            w:r       → 提取文本 + 粗体/斜体标记
│   │            m:oMath   → [XSLT] OMML → MathML → LaTeX → $$...$$
│   │            m:oMathPara → 同上 (display 公式)
│   │            w:drawing → 提取 a:blip/@r:embed → image_map → ![](path)
│   │            w:object  → [VML命名空间] v:imagedata/@r:id → image_map → ![](path)
│   │                        或 o:OLEObject/@ProgID 区分类型
│   │
│   └── w:tbl → 遍历 w:tr/w:tc
│                 └── w:tc 内部 w:p → parse_paragraph_children() [递归]
│                     (表格内公式、图片一并保留)
│               最终 → Markdown pipe table

Phase 3: 输出
├── 拼接 Markdown 片段
└── 写入 output_md_path
```

## 5. 已知限制

1. **MathType 公式不可编辑**：导出为 PNG，无法在 Markdown 中数学编辑
2. **OMML → LaTeX 覆盖率**：XSLT 覆盖 Office 标准公式集合，极端复杂构造可能转换不佳
3. **仅支持 .docx**：不处理 .doc 格式
4. **OLE 嵌套不递归**：OLE 中嵌套 OLE 的场景不做深层处理
5. **WMF 渲染依赖 Windows**：Linux 环境需降级到 Inkscape 或保留原文件
