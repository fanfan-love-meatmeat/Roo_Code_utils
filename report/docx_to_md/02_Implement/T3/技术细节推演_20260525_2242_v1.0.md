### 1.2 规范对照：EQN_PREFS 真实结构

Wiris 规范明确 EQN_PREFS 为**变长**结构：

```
EQN_PREFS (0x12):
  tag(1) | options(1)
  ├─ Dimension Array (size values):
  │    size_array_count(1)          ← 尺寸项数量 (e.g. 8)
  │    nibble_array(variable)       ← 每个尺寸编码为 nibble 压缩的字符串
  │                                  ("12 points", "58%", "42%", ...)
  └─ Style Definition Array:
       style_count(1)               ← 样式项数量
       style_defs(variable)         ← 每个样式: FONT_DEF_index(uint16, 0=未用)
                                     + optional char_style(1, 仅当 index≠0)
```

**关键发现**: 当前 `_parse_eqn_prefs()` 硬编码跳过 80 字节，而实际长度取决于 size_array_count 内各值的 nibble 编码长度。对于 8 个尺寸项，每项 8-12 nibbles，总计约 64-96 nibbles ≈ 32-48 bytes。加上 style 定义（7-12 个 × 2-3 bytes = 14-36 bytes），EQN_PREFS 总长可达 50-90+ bytes。

我们的文档有 8 个尺寸项（`08` at 0x70），nibble 数组约消耗 80 字节，style 定义区从 0xBF 开始。**80 字节跳过刚好卡在 style 定义区边界**，但未消费 style 定义。

### 1.3 当前短乱码的成因链

```
EQN_PREFS@0x6E → skip 80B → cursor@0xBF → 此处仍是 style_definitions!
  → 0x0C (style_count=12) 被当作 SUB2 类型标签
  → 后续 style_def 字节被当作 LINE/CHAR/TMPL 解析
  → 输出: Ȃā\langle䎃\rangle
```

**本质**: 游标落在了 EQN_PREFS 的 style definition 区，而非真正的方程 AST。

### 1.4 CTO 指令合理性评估

| CTO 指令 | 与规范一致性 | 评估 |
|----------|-------------|------|
| Directive 1: Pre-AST 嗅探与消化 | ✅ 合理 | 在 `_skip_header` 后、`_parse_line` 前插入预处理器，消费 SIZE/style 记录 |
| Directive 2: 动态定界与容错 | ✅ 必要 | 硬编码 80B 跳过必须替换为 count-driven 解析 |
| Directive 3: AST 防污染隔离 | ✅ 防御 | CHAR 节点输出控制字符/CJK 私用区时记录告警 |

**综合评定**: CTO 指令完全正确。Pre-AST 层的引入是解决当前短乱码的最低侵入度方案。核心策略是**修复 EQN_PREFS 的变长解析**，使得 cursor 精准落在方程 AST 起点。

### 2.2 修改详情

#### 修改 A：重写 `_parse_eqn_prefs()` — 变长解析

**变更前**（死板 80B 跳过）:
```python
def _parse_eqn_prefs(self):
    self.pos = min(len(self.data), self.pos + 80)
```

**变更后**（count-driven 变长解析）:
```python
def _parse_eqn_prefs(self):
    # Step 1: 读取 size_array_count，消费 nibble 数组
    size_count = self._read_byte()
    # nibble 数组: 每 2 nibbles = 1 byte，但编码长度不定
    # 安全策略: 逐尺寸扫描，直至消费所有 nibble 项
    self._consume_dimension_array(size_count)

    # Step 2: 读取 style_count，消费 style 定义
    style_count = self._read_byte()
    self._consume_style_definitions(style_count)
```

**`_consume_dimension_array`** 策略: 
Wiris 规范中 dimension array 的 nibble 编码以每项的终止模式为界。稳妥方案：按 count 值循环读取每项，通过 nibble 编码规则（高 nibble 0x0-0x7 为数字，0x8-0xF 为标记）逐项消费直到遇到终止符（0xF?）或超出 count。

备选简化方案：尺寸项的字符编码以 null nibble (0x00) 或高位标记 (0xF0) 分隔，先读 size_count 个 null-terminated-like 的 nibble 块。

**`_consume_style_definitions`** 策略:
每项读取一个 uint16_le (FONT_DEF index)。若值 ≠ 0，紧跟 1 byte char_style。循环 style_count 次。

#### 修改 B：新增 `_consume_size_prefix()` — Pre-AST 预处理器

在 `parse()` 方法中，`_skip_header()` 之后、`_parse_line()` 之前插入：

```python
def parse(self) -> str:
    self.pos = 0
    self._skip_header()

    if self.pos >= len(self.data):
        return ''

    # === Pre-AST: 消费方程数据区 SIZE 前缀 ===
    self._consume_size_prefix()

    parts = self._parse_line()
    return ''.join(parts)
```

**`_consume_size_prefix`** 策略:
循环窥探当前字节 tag：
- 若 `tag ∈ {0x09 (SIZE), 0x0A (FULL), 0x0B (SUB), 0x0C (SUB2), 0x0D (SYM), 0x0E (SUBSYM)}`：
  - SIZE (0x09): 调用 `_parse_size_record()` 消费 1 条 SIZE 记录
  - FULL/SUB/SUB2/SYM/SUBSYM: 仅 skip 1 byte (typesize 标签无附加数据)
- 若 `tag ∈ {0x01 (LINE), 0x02 (CHAR), 0x03 (TMPL), 0x00 (END)}`：break，方程 AST 开始

此 Pre-AST 处理器同时解决：
1. EQN_PREFS 泄漏的 style definition 字节（如果 `_parse_eqn_prefs` 修复后仍有残留）
2. 方程数据区合法的 SIZE/FULL/SUB... 前缀定义

#### 修改 C：CHAR 节点防污染断言

在 `_parse_line()` 的 CHAR 处理分支中增加非侵入式断言：

```python
elif record_type == CHAR:
    # ... 现有 CHAR 解析逻辑 ...
    ch = self._read_char_value(typeface, options)

    # 防污染断言: 若产生控制字符/私用区字符，说明游标可能错位
    if ch:
        cp = ord(ch[0]) if len(ch) == 1 else 0
        if (0x0000 <= cp <= 0x001F) or (0xE000 <= cp <= 0xF8FF):
            log.debug(f"[AST_POLLUTION] CHAR at pos={orig_pos} produced U+{cp:04X}, "
                      f"typeface={typeface}, nearby bytes: {self.data[self.pos-4:self.pos+4].hex()}")
    parts.append(ch)
```

## 3. 逻辑流转

### 3.1 修复前流程

```
parse()
  ├─ _skip_header()                    → cursor 到 header 尾
  │    正确跳过 FONT_DEF/ENCODING_DEF/EQN_PREFS(80B)  → cursor @ style_defs 中间 ❌
  ├─ _parse_line()                     → 从 style_def 字节开始
  │    style_def bytes 被当作 CHAR/TMPL → 短乱码: Ȃā\langle䎃\rangle
  └─ 263 公式输出: 短乱码
```

### 3.2 修复后流程

```
parse()
  ├─ _skip_header()                    → cursor 到 header 尾
  │    EQN_PREFS 变长解析: nibble_array(80B) + style_defs(36B) → cursor @ 方程数据边界 ✅
  ├─ _consume_size_prefix()            → Pre-AST 消费 SIZE/FULL/SUB/...
  │   消费: SUB2, FULL, ... 类型标签  → cursor @ 第一个 LINE/CHAR/TMPL ✅
  ├─ _parse_line()                     → 从正确的 AST 根节点开始
  │   解析 LINE → CHAR('x') → TMPL(subscript) → CHAR('c') → END
  │   输出: x_c 或 $$x_c$$ ✅
  └─ 263 公式输出: 正确 LaTeX 结构
```

### 3.3 预期公式对应

| 原文位置 | 预期公式 | 预期 LaTeX |
|----------|----------|------------|
| S1 作物行中心点 | x_c | `$x_c$` / `$$x_c$$` |
| S1 RGB 相机视野中心 | x_0 | `$x_0$` |
| S1 偏差值 | Δx | `$\Delta x$` |
| S2 人体中心点 | P_center | `$P_{center}$` |
| S3 俯仰角/偏航角 | θ_p, θ_y | `$\theta_p$`, `$\theta_y$` |
| S4 投影距离 | d_proj | `$d_{proj}$` |
| S4 深度值 | Z | `$Z$` |

## 4. 防御机制

| 防御层 | 实现位置 | 机制 |
|--------|----------|------|
| L0: EQN_PREFS 变长解析 | `_parse_eqn_prefs()` | count-driven 消费，替代 80B 硬编码 |
| L1: Pre-AST SIZE 前缀消费 | `_consume_size_prefix()` | 在 `_parse_line` 前消费 SIZE/FULL/SUB/SUB2/SYM/SUBSYM |
| L2: Nibble 数组边界保护 | `_consume_dimension_array()` | 计数循环 + min() 上界截断 |
| L2: Style 定义边界保护 | `_consume_style_definitions()` | 计数循环 + 位置断言 |
| L3: CHAR 防污染 | `_parse_line()` CHAR 分支 | 控制字符/CJK 私用区检测 + debug 日志 |

## 5. 风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| Nibble 数组格式因 MathType 版本差异而解析失败 | 低 | 中 | 保留 80B fallback；维度解析失败时降级到 heuristic skip |
| 部分公式 SIZE 前缀后无 LINE → 死循环 | 极低 | 低 | `_consume_size_prefix` 有 step 计数器上限 |
| 修复后公式识别率未达 100% | 中 | 中 | 公式编码（MTCode/Symbol/MTExtra）是独立问题，不影响本次 SIZE 表修复 |
