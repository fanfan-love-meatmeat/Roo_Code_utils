## 3. 实施过程复盘：走过的弯与踩过的坑

本次"双字节包头"看似简单的全局替换，实际执行中经历了 5 轮迭代修正，每一轮都暴露了协议理解与工程假设之间的偏差。

### 弯路 1：盲目全量双字节 → END 吞噬 options
**错误**: 对 `_parse_line` 中**所有** record 无条件读取 `tag + options`，包括 END (0x00)。
**现象**: 263 公式全部输出空字符串。END 的 options 字节（`00`）被读走后，游标错位，上一层 LINE 提前终止。
**教训**: END 是终结符，无 payload 也无 options。必须在 `tag == END` 之后、`options = _read_byte()` 之前提前 `return`。

### 弯路 2：`_consume_size_prefix` 窥探不消费 → 仍全空
**错误**: prefix 中对 LINE/CHAR/TMPL 仅 `peek`（`self.data[self.pos]`）不消费，但继续对 FULL/SIZE 读取 `tag + options`。FULL (0x0A) 的 "options" 字节恰好是下一记录的 LINE tag (0x01)，导致整个 LINE 被当作 FULL 的 options 吃掉。
**现象**: 仍全空。因为公式 #0 的数据 `0A 01 00...` 中，`01` 被当作 FULL 的 options，后续 `00` 触发了 END。
**教训**: FULL/SUB/SUB2/SYM/SUBSYM 等 typesize 标签在此文档中的物理形态并非独立的 `tag + options` 记录 —— 它们与后续 LINE 的 tag 字节重叠。根本原因是 `_skip_header` 的方程数据边界有 1 字节偏移。

### 弯路 3：边界偏移与 `+1` 对齐
**错误**: 方程数据边界在 `_skip_header` 结束时多出 1 字节（pos 216 的 `0A` 实际上是 style 定义区的残留，不属于方程数据）。
**修正**: `_skip_header` 尾部追加 `if tag in (FULL, SUB, ...): self.pos += 1`，将游标推进到正确的方程数据起始位。
**效果**: LINE/CHAR 开始产生正确输出 (`'o'`, `'\\Chi'`)。

### 弯路 4：TMPL `03 71` 的 OOB 崩溃
**错误**: 双字节包头对 TMPL 同样生效：`tag=03, options=71`。options `71` 的高位设置了 `xfLMOVE` 标志，导致 TMPL 分支去读取 nudge 字节（2 字节），后续 selector 读取越界。
**现象**: 28 公式报错 `Unexpected EOF`。
**根因**: 该文档中 TMPL 的 options **仍在 tag 高 nibble**（`03 >> 4 = 0`），而非独立字节。`71` 是 TMPL 的 selector 字节，不是 options。
**修正**: TMPL 分支回退 1 字节（`self.pos -= 1`），改用 `options = tag >> 4`（nibble-packed）。

### 弯路 5：混合编码的协议现实
**最终方案**: 本文档的 MTEF v5 实现存在混合编码 —— **LINE/CHAR 使用独立 options 字节，TMPL 使用 nibble-packed options**。这与 Wiris 官方示例（所有记录均为独立 options）不完全一致，系 MathType 7.x 特定版本的私有变体。

**最终架构**: 
- `_parse_line`: `tag=_read_byte()` + `options=_read_byte()`（LINE/CHAR/SIZE 等）
- TMPL 分支: `self.pos -= 1` + `options = tag >> 4`（回退到 nibble）
- `_consume_size_prefix`: 仅处理 SIZE(0x09)，窥探 LINE/CHAR/TMPL/END/FULL 即停手
- `_skip_header`: 标准记录 `tag` 直接比较 + 收尾 1 字节对齐修正

### 经验总结

| # | 教训 |
|---|------|
| 1 | **协议文档不等同于实际实现** — Wiris 示例全用独立 options，但手稿可能有混合编码 |
| 2 | **END 是特殊终结符** — 无 payload 无 options，必须优先判断 |
| 3 | **边界偏移是常见兼容问题** — style 定义区与方程数据之间可能有 1-2 字节 padding |
| 4 | **逐记录类型验证** — 不能假设一种编码适用于所有记录类型 |
