## 1. 检索与论证

### 1.1 Wiris 官方证据

Wiris MTEF v5 规范 byte-by-byte 示例（0x011A 处）：

```
| 282 | 0x011A | 136, 0x88 | typeface: 136 - 128 = 8 (Number style)
| 283 | 0x011B | 50, 0x32, '2' | MTCode value: 0x0032 ('2')
```

以及 0x0107 处：

```
| 263 | 0x0107 | 131, 0x83 | typeface: 131 - 128 = 3 (Variable style)
| 264 | 0x0108 | 97, 0x61, 'a' | MTCode value: 0x0061 ('a')
```

**关键结论**:
- **typeface ≥ 128 → MTCode 单字节编码**。字符值读取为 1 字节（非 uint16）。
- typeface - 128 = FONT_DEF 索引，指向具体字体。
- MTCode 是一个自定义代码页，不同字体映射不同（如 Number 风格 0x32 → '2'，Variable 风格 0x61 → 'a'）。

### 1.2 当前 Bug 成因

当前 `_read_char_value` 始终按 `options & mtefOPT_CHAR_ENC_CHAR_8` 决定单/双字节，**完全忽略 typeface ≥ 128 的 MTCode 语义**：

```python
def _read_char_value(self, typeface: int, options: int) -> str:
    if options & mtefOPT_CHAR_ENC_CHAR_8:  # 根据 options 标志
        char_code = self._read_byte()       # 1 byte
    else:
        char_val = self._read_uint16_le()   # 2 bytes ← 错！typeface≥128时应为1 byte
    return self._char_to_latex(char_val, typeface)
```

对于我们的文档：
- 数据 `02 00 83 6F` → typeface=0x00（错误读取）, char_val=0x6F83（uint16 LE）
- 应该解析为：typeface=0x83, char_val=0x6F（MTCode 单字节）

**typeface=0x00 的原因**: `00` 字节被当作 typeface，而 `83` 被当作 char_val 低字节。正确解析应跳过中间的 `00`（它是 style 定义残留字节或空填充），或重新对齐读取。

### 1.3 CTO 指令评估

| 指令 | 与规范一致性 | 评估 |
|------|-------------|------|
| 指令一：Typeface 联动映射 | ✅ 完全正确 | typeface ≥ 128 时路由到 MTCode 单字节 → 字体对照表 |
| 指令二：PUA 启发式对齐 | ✅ 防御性 | 私用区字符作为降级兜底 |
| 指令三：状态机对齐断言 | ✅ 必要 | Hex dump 日志用于后续微调 |

### 1.4 已知 MTCode 映射关系（部分）

根据 transpect/fontmaps 开源数据和 Wiris 编码定义：

```
MTCode (编码索引=1): Times New Roman, Courier 等文本字体 → 标准 ASCII/Windows-1252
Symbol (编码索引=3): 符号字体 → 希腊字母 + 数学符号
  MTCode 0x61='α'(alpha), 0x62='β'(beta), 0x67='γ'(gamma), 0x64='δ'
  MTCode 0x70='π'(pi), 0x71='θ'(theta), 0x73='σ'(sigma), 0x77='ω'
  MTCode 0x41='Α'(Alpha), 0x44='Δ'(Delta), 0x50='Π'(Pi), 0x53='Σ'
  MTCode 0x2A='×'(times), 0xB8='÷'(divide), 0xB3='≥'(ge)
MT Extra (编码索引=4): 扩展数学符号 → 各种运算符
  （完整表见 transpect/MT_Extra.xml）
```

### 2.2 修改详情

#### 修改 A：重构 `_read_char_value()` — typeface ≥ 128 单字节读取

```python
def _read_char_value(self, typeface: int, options: int) -> str:
    if typeface >= 128:
        # MTCode: typeface - 128 = FONT_DEF index, char is single-byte
        char_val = self._read_byte()
    elif options & mtefOPT_CHAR_ENC_CHAR_8:
        char_val = self._read_byte()
    else:
        char_val = self._read_uint16_le()
    return self._char_to_latex(char_val, typeface)
```

#### 修改 B：新增 `_mtcode_to_latex()` — 按字体编码映射表

```
核心映射表结构（三张子表）:

MTCode (text fonts: Times New Roman, Courier):
  0x20-0x7E → 标准 ASCII 直通
  0x80-0xFF → 扩展 Windows-1252

Symbol Encoding (font: Symbol):
  0x20-0x7E → 符号字体的希腊/数学映射
  关键映射: 见 1.4 节

MT Extra Encoding (font: MT Extra):
  0x20-0x7E → 扩展数学运算符
```

实现策略：
- 创建 `MTOKEN_SYMBOL` 静态字典（200+ 条目），覆盖 Symbol 字体全部映射
- `_mtcode_to_latex(char_val, font_idx)` 根据 font_map[font_idx] 的字体名路由到对应子表
- 未命中返回 `[MTCode:0x{char_val:02X}]` 占位符

#### 修改 C：`_char_to_latex()` — 增加 typeface ≥ 128 的 MTCode 路由

```python
def _char_to_latex(self, char_val: int, typeface: int) -> str:
    if typeface >= 128:
        font_idx = typeface - 128
        return self._mtcode_to_latex(char_val, font_idx)
    # 原有逻辑不变（typeface < 128 的标准处理）
    ...
```

#### 修改 D：ASCII 字符直通优化

MTCode 对于 Text 风格（font_idx 指向 Times New Roman 等）直接映射 ASCII：
- MTCode 0x20-0x7E → 对应 ASCII 字符
- 无需查表，直接 `chr(char_val)` 再转义特殊字符

## 3. 逻辑流转

### 3.1 修复前

```
CHAR(tag=02) → typeface=0x00(误) → char_val=0x6F83(uint16) → _char_to_latex
  → 0x6F83 >= 0x100 → chr(0x6F83) = '澃' ❌
```

### 3.2 修复后

```
CHAR(tag=02) → typeface=0x83(正确) → MTCode(char_val=0x6F, 1 byte)
  → _char_to_latex(char_val=0x6F, typeface=0x83)
  → typeface ≥ 128 → font_idx = 3
  → _mtcode_to_latex(0x6F, font_idx=3)
  → font_map[3] = "Symbol" → Symbol 编码表
  → MTCode 0x6F in Symbol = ? (查表)
  → 输出: $\theta$ 或期望的 LaTeX ✅
```

### 3.3 预期公式对应

| 当前输出 | 原始字节 | MTCode | 字体 | 预期 LaTeX |
|----------|----------|--------|------|------------|
| `澃` (U+6F83) | `83 6F` | 0x6F | Symbol | 待查表确定 |
| `䎃` (U+4383) | `83 43` | 0x43 | Symbol | `\Gamma` 或 `C` |
| `䒃` (U+4483) | `83 44` | 0x44 | Symbol | `\Delta` |
| `劃` (U+5283) | `83 52` | 0x52 | Symbol | `P` 或 `\Pi` |

CTO 推测：`뢄` → `\theta_p`, `햄` → `\theta_y`。需在映射表实现后验证。

## 4. 防御机制

| 防御层 | 实现 |
|--------|------|
| typeface 门控 | `typeface >= 128` 强制单字节 MTCode，不受 options 标志干扰 |
| 映射表未命中 | 返回 `[MTCode:0x{char_val:02X}]` 占位符，不输出生僻 CJK |
| 字体名 fallback | font_map 未命中时默认使用 Symbol 编码表 |
| PUA 启发式降级 | `char_val` 落入 0xE000-0xF8FF 时记录告警并尝试回退 |

---

## 5. 风险矩阵

| 风险 | 概率 | 缓解 |
|------|------|------|
| Symbol 编码表条目不全 | 中 | 覆盖高频 128 条目，其余用 `[MTCode:0xNN]` 占位 |
| 非 Symbol 字体映射未覆盖 | 低 | MT Extra 编码表从 transpect 开源数据提取 |
| typeface=0x00 字符被误伤 | 低 | typeface >= 128 门控不影响 typeface < 128 的标准路径 |
