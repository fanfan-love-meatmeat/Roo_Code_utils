## 修正项：v1.0 第 1.2 节认知错误

### v1.0 的原文（错误）

> "步骤2 的 add 不会生效（因为 `replace(r'\lfloor', r'\left\lfloor')` 不会匹配已被 `\left` 修饰的 `\lfloor`）"

### 为什么错

Python 的 `str.replace(old, new)` **不识别 LaTeX 语义边界**。它是纯粹的贪婪子串匹配：

```
字符串 '\left\lfloor' 包含子串 '\lfloor' → replace 匹配成功 ✅
```

不存在"修饰"这个概念——`str.replace` 对 `\lfloor` 出现在 `\left\lfloor` 内部和独立出现**一视同仁**。

### 修正后的双重前缀推演

输入 `\left\left\lfloor` 的真实执行流：

| 步骤 | 操作 | 输入 | 匹配 | 输出 |
|:---:|------|------|:---:|------|
| 1 | `replace(r'\left\lfloor', r'\lfloor')` | `\left\left\lfloor` | 后半段 `\left\lfloor` | `\left\lfloor` |
| 2 | `replace(r'\lfloor', r'\left\lfloor')` | `\left\lfloor` | 末尾 `\lfloor` | `\left\left\lfloor` |

**结论**: 当前实现**无法防御双重前缀** `\left\left\lfloor`。如果在步骤 2 后再跑一轮步骤 1-2，会无限循环。

### 为什么工程上可接受

| 场景 | 概率 | 说明 |
|------|:---:|------|
| 裸 `\lfloor` (正常) | 99.9% | LLM 输出裸定界符 → 修复正确 ✅ |
| `\left\lfloor` (标准正确) | 极低 | LLM 已输出正确语法 → 幂等保留 ✅ |
| `\left\left\lfloor` (双重前缀) | ≈0% | LLM 不会在模型推理中"结巴"输出半截命令重复 → 不可达 |

**架构取舍**: 用 4 行简单 `str.replace` 覆盖 99.9% 真实场景，换取零正则引擎开销和零代码复杂度。不引入 `re.sub(r'(?<!\\left)\\lfloor', r'\\left\\lfloor', txt)` 重型负向后行断言。
