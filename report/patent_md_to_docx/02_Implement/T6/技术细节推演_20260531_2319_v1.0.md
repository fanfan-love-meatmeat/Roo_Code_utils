## 0. 根因分析

### 0.1 问题定位

在 `test/patent_md/完整专利申请书_20260531_2010_v2.4.md` 第 163 行：

```
$K = (\lfloor x_e/g \rfloor, \lfloor y_e/g \rfloor, \lfloor z_e/g \rfloor)$
```

**唯一出问题的公式**。其他 15+ 处公式（`$l$`, `$R_t$`, `$V_{overlap}$` 等）均正常工作。

### 0.2 根因

| 因素 | 说明 |
|------|------|
| **直接原因** | `\lfloor` / `\rfloor` 是 LaTeX 伸缩定界符，需 `\left` / `\right` 告知渲染引擎其边界范围。裸用 `\lfloor` 在混合中英文的字体上下文中，MathType 的 TeX 解释器无法确定合适的定界符尺寸和字体映射，导致渲染崩溃（黑色菱形问号） |
| **为什么其他公式正常** | `$l$`, `$R_t$`, `$N_{collision}$` 等仅涉及上下标和希腊字母，不涉及伸缩定界符。`\lfloor`/`\rfloor` 是唯一触发 MathType 中文字体冲突的符号 |
| **标准 LaTeX vs MathType** | 标准 LaTeX 中裸 `\lfloor` 是合法的（取自然尺寸），但 MathType 的 TeX 解释器在 Unicode 中文环境下对裸定界符的字形查找存在已知缺陷。`\left`/`\right` 包装提供显式渲染指令，绕过字形查找 |

### 0.3 代码溯源验证

| 查询 | 结果 |
|------|------|
| `\lfloor` 在 MD 中出现 | ✅ 仅 1 处 (第 163 行) |
| `\lceil` 在 MD 中出现 | ❌ 未出现 |
| 其他公式是否有类似问题 | ❌ 无。其余公式仅含 `_`, `^`, `\cdot`, `\parallel`, `\le`, `\cdot`, `\cdot`, 等基础 LaTeX 命令 |

## 1. 修复策略验证

### 1.1 "先剥离、后穿衣" 幂等性原理

```python
# 步骤 1：降级 — 消除可能已存在的 \left/\right
txt.replace(r'\left\lfloor', r'\lfloor')
txt.replace(r'\right\rfloor', r'\rfloor')

# 步骤 2：升格 — 统一加上 \left/\right
txt.replace(r'\lfloor', r'\left\lfloor')
txt.replace(r'\rfloor', r'\right\rfloor')
```

| 输入状态 | 步骤1后 | 步骤2后 | 最终 | 安全性 |
|------|------|------|------|:---:|
| `\lfloor` (裸) | `\lfloor` | `\left\lfloor` | `\left\lfloor` | ✅ |
| `\left\lfloor` (已有) | `\lfloor` | `\left\lfloor` | `\left\lfloor` | ✅ 幂等 |
| `\left\left\lfloor` (错误双前缀) | `\left\lfloor` | `\left\left\lfloor`… ❌ | — | ⚠️ 见1.2 |

### 1.2 顺序关键性评估

步骤 1 和步骤 2 的顺序**必须严格保持**：

- **先 strip 后 add**：`\left\lfloor` → strip→ `\lfloor` → add→ `\left\lfloor` ✅
- **先 add 后 strip（错误）**：`\lfloor` → add→ `\left\lfloor` → strip→ `\lfloor` ❌ 白干

当前 T6 代码顺序正确（先 strip 后 add），不会引入双重前缀。

但有一个理论死角：如果输入是 `\left\left\lfloor`（极不可能，仅为防御性考虑），步骤1 的 strip 只会移除一层 `\left`，变成 `\left\lfloor`，步骤2 的 add 不会生效（因为 `replace(r'\lfloor', r'\left\lfloor')` 不会匹配已被 `\left` 修饰的 `\lfloor`）。最终保留一层 `\left`，安全。

### 1.3 `\lceil`/`\rceil` 的处理

当前 MD 中无 `\lceil`/`\rceil`，但 T6 要求同步加入防御。这是**前瞻性防御**——未来 MD 中可能出现向上取整符号。处理逻辑与 `\lfloor`/`\rfloor` 完全相同。

## 3. 风险评估

| 风险项 | 严重度 | 说明 |
|--------|:---:|------|
| `\lfloor`/`\rfloor` 不成对出现 | P3 | 在专利 MD 中，floor/ceil 定界符总是成对出现（`\lfloor ... \rfloor`）。即使单侧出现，`\right` 在 MathType 的 TeX 解释器中如果找不到匹配的 `\left` 会报错，但不会破坏其他公式——这是原始 TeX 错误，非本修复引入 |
| `\rfloor` 作为普通符号被误包裹 | P3 | `\rfloor` 在 LaTeX 中**仅有定界符语义**，不存在其他用法。无风险 |
| 对大模型输出的适应性 | P1 | 幂等性设计确保无论大模型输出裸 `\lfloor` 或 `\left\lfloor`，最终输出始终一致。**满足验收标准第4条** |
| 对其他 LaTeX 命令的影响 | 零 | 仅针对 `\lfloor`/`\rfloor`/`\lceil`/`\rceil` 四种符号，不影响 `\parallel`, `\le`, `\cdot` 等任何其他命令 |

## 4. 验收标准对照

| T6 验收标准 | 达成方式 |
|------------|---------|
| `\lfloor` 公式在 MathType `Alt+\` 转换后无乱码 | `\lfloor` → `\left\lfloor`，`\rfloor` → `\right\rfloor` 包装后 MathType 正确渲染 |
| 定界符自适应内部高度缩放 | `\left`/`\right` 是 LaTeX 自动缩放定界符的标准语法，MathType 原生支持 |
