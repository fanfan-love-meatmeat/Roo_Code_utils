## 第二部分：核心挑战与架构决策

### 2.1 挑战 A：MD 中无 `# heading` — 标题识别策略

**问题**: T1 假设 MD 使用 `# heading` 语法，但实际专利 MD 不使用。所有"标题"均为扁平化纯文本/粗体段落。

**决策**: 采用**启发式上下文识别**替代 AST heading token：

```python
# 节标题识别规则 (按优先级)
def classify_section(text: str) -> str:
    """返回: 'section_title' | 'sub_title' | 'claim' | 'numbered_para' | 'normal'"""
    # 1. 权利要求行
    if re.match(r'^\d+\.\s', text):
        return 'claim'
    # 2. 说明书编号段落
    if re.match(r'^\[(\d{4})\]', text):
        return 'numbered_para'
    # 3. 节标题: 全空格分隔的连续汉字 (如 "权 利 要 求 书")
    if re.match(r'^[\u4e00-\u9fff]\s[\u4e00-\u9fff]', text) and len(text) < 30:
        return 'section_title'
    # 4. 粗体子标题: **XXX** 独占整行
    if re.match(r'^\*\*[^*]+\*\*$', text.strip()):
        return 'sub_title'
    return 'normal'
```

### 2.2 挑战 B：`** ** ` (bold spaces) 污染

**问题**: 上游 MD 中存在 `**[0001]** ** ** 本发明...` 这样的模式 —— `** **` 是两个空格被标为粗体。在 DOCX 中，这会产生不必要的加粗空格 Run。

**解决**: 遍历 inline children 时，跳过 `content` 为纯空格（或仅含空格）的 `strong` token 内部的 `text` token：

```python
def is_whitespace_strong(token):
    """判断 strong token 是否仅包含空白字符"""
    if token.type == 'strong_open':
        # 找到下一个 text token
        next_t = ...
        if next_t and next_t.content.strip() == '':
            return True
    return False
```

### 2.3 挑战 C：字体族的 XML 级操作

**问题**: python-docx 的 `run.font.name` 仅设置西文字体。中文字体需通过 XML 操作设置 `w:eastAsia` 属性。

**正解** (已在现有 `md_to_docx.py` 中验证):

```python
from docx.oxml.ns import qn

def set_run_font(run, cn_font='宋体', en_font='Times New Roman', size=Pt(12)):
    """同时设置中英文字体"""
    run.font.name = en_font
    run.font.size = size
    rPr = run._element.get_or_add_rPr()
    rFonts = rPr.find(qn('w:rFonts'))
    if rFonts is None:
        rFonts = rPr.makeelement(qn('w:rFonts'), {})
        rPr.insert(0, rFonts)
    rFonts.set(qn('w:eastAsia'), cn_font)
    rFonts.set(qn('w:ascii'), en_font)
    rFonts.set(qn('w:hAnsi'), en_font)
```

**联网查证**: `w:eastAsia` 是 python-docx 社区广泛使用的扩展方法（GitHub PR #329 讨论确认），虽未被官方 API 吸收，但功能稳定。

### 2.4 挑战 D：`[0001]` 编号提取的边界情况

**需防御的模式**:
- `**[0001]**` (正确格式)
- `[0001]` (不加粗 — 降级处理)
- `[0001a]` (编号变体 — 按原样处理，不视为标准编号)
- 正文中引用其他段落的 `[0001]` (如"参见 [0001] 段") — 需通过**位置约束**排除：仅在行首（去除前导空格后）的 `[NNNN]` 才视为段落编号

### 2.5 挑战 E：Markdown `---` 分割线处理

目标 MD 中用 `---` 分隔"权利要求书"、"说明书"、"具体实施方式"三大块。`markdown-it-py` 将其解析为 `hr` token。

**决策**: `hr` → 在 DOCX 中生成分页符 (`doc.add_page_break()`)，使三大块各自独立起页（参考模板中也是各自分页）。

### 4.1 解析器架构

```

    │
    ▼

    ▼
```
