### 逐项验证结论

**异常表象 A (ModuleNotFoundError) → ✅ 诊断正确**

MCP stdio 传输规范要求 server 启动后立即完成 `initialize` → `initialized` 握手。server.py 顶层 `import` 任何未安装的依赖都会在启动瞬间触发 `exit(1)`，客户端收到进程终止信号后报告 `-32000: Connection closed`。

D1 建议的修复路径「裸机启动 + 逐包 pip install」是该类问题的标准排查流程。

---

**异常表象 B (Stdout 污染) → ✅ 诊断正确，且为 -32000 最高频根因**

MCP 规范原文（transports 章节）：

> *"The server MUST NOT write anything to its `stdout` that is not a valid MCP message."*

社区统计（agenticmarket.dev / mcpstack.org）均将 stdout 污染列为 -32000 的首位诱因。典型案例：

- `powertools-mcp` #245: `console.log` 写入 stdout → `serde error data did not match any variant of untagged enum JsonRpcMessage`
- `wikipedia-mcp` #4: 全局 `print()` → stdio 模式下完全不可用

D1 建议的「全局检索 print 语句 → 删除或 `file=sys.stderr`」与社区最佳实践完全一致。

---

**异常表象 C (Windows 编码页) → ✅ 诊断正确**

Windows 中文环境默认使用 cp936 (GBK) 编码。MCP 协议要求纯 UTF-8 JSON-RPC 流。当 server 输出含中文的 JSON 时，stdout 编码不匹配会导致客户端解析失败。

官方 Issue [modelcontextprotocol/servers#65](https://github.com/modelcontextprotocol/servers/issues/65) 明确确认 Windows 下需设置 `PYTHONIOENCODING=utf-8`。PEP 540 规定的 `PYTHONUTF8=1` 是强制 UTF-8 模式的标准环境变量。

D1 建议的 `PYTHONIOENCODING=utf-8` + `PYTHONUTF8=1` 组合是 Windows 下 MCP 服务的必要配置。
