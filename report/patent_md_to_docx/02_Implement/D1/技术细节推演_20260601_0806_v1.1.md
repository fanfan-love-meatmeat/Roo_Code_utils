## 1. 检索与论证

### 检索源

| 来源 | 关键结论 |
|---|---|
| **MCP 官方规范** (transports) | stdout 必须仅输出合法 JSON-RPC 消息，任何 `print()` 污染直接导致 -32000 |
| **GitHub #65** (modelcontextprotocol/servers) | Windows 需设 `PYTHONIOENCODING=utf-8` |
| **PEP 540** | `PYTHONUTF8=1` 为强制 UTF-8 模式标准开关 |
| **nvie.com** (Pin Your Packages) | 生产级 `requirements.txt` 必须用 `==` 精确锚定版本，绝不用裸名或范围 |
| **IBM Data Science Best Practices** | 固定版本格式是部署应用的行业最佳实践，保证确定性构建 |
| **Xygeni** (Hidden Dangers) | 未锚定版本的依赖是供应链攻击的第一入口 |
| **Python.org 社区讨论** | 广泛共识：精确版本 → 可复现；裸名/范围 → 漂移风险 |

### 1.3 架构师修正意见验证

#### 漏洞 1: D1.md 破坏 SSOT 原则 → ✅ 合理

社区共识：在多文件项目中，依赖清单只能有一个权威源。D1.md 作为排查指引，不应再列出具体包名——任何重复列表都会随时间漂移。

**修正**: D1.md L24 改为执行 `pip install -r requirements.txt`，剥离其维护依赖名单的职责。

#### 漏洞 2: 裸奔的 requirements.txt 缺乏版本锚点 → ✅ 合理

检索确认这是 Python 生态的普遍共识：

- **nvie.com**: "Don't ever use `lxml` or `lxml>=2.2.0`. Instead, pin them: `lxml==2.3.4`"
- **IBM Best Practices**: "This is considered a best practice for deploying applications, and ensures an explicit runtime environment with deterministic builds."
- **Xygeni**: "Dependency pinning is more than a best practice. It's your first line of defense against supply chain attacks."

**修正**: 所有依赖使用 `==` 精确锚定当前稳定版本。

## 4. 防御机制

| 层级 | 机制 | 说明 |
|---|---|---|
| **SSOT** | 单文件权威 | `requirements.txt` 为依赖清单唯一真相源，README/D1/env_backup 仅引用不复制 |
| **版本锚定** | `==` 精确锁定 | 杜绝 `pip install` 拉取最新版导致的静默 Breaking Change |
| **尽职剔除** | Pillow 移除 | 全量 import 扫描确认无直接引用，避免冗余依赖扩大攻击面 |
| **过期清理** | env_backup 同步 | 移除 pix2tex/torch 等已弃用信息，防止误导 |
