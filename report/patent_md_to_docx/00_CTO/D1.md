**任务目标**：定位并修复 `roo_utils` MCP 服务器在 Kilo Code / Roo Code 中抛出 `Connection closed` 的底层崩溃原因。
**测试环境**：Windows 11 / Conda (`roo_mcp` 独立环境)
**目标文件**：`D:/FJL/Projects/Kilo_Code_Gobal_Settings/Roo_Code_utils/server.py`

## 1. 核心问题定位（裸机测试）

当前 GUI 掩盖了真实的 Python 报错堆栈。测试团队需**完全脱离 Kilo/Roo Code**，直接在 PowerShell 中执行以下原生启动命令：

```powershell
# 强制通过目标 Conda 环境的 Python 解释器启动 Server
C:/Users/98514/.conda/envs/roo_mcp/python.exe D:/FJL/Projects/Kilo_Code_Gobal_Settings/Roo_Code_utils/server.py
```

执行后，请严格对照以下三种崩溃表象进行修复：

### 异常表象 A：依赖缺失闪退 (ModuleNotFoundError)
**特征**：终端直接抛出 `No module named 'xxx'`。由于 `server.py` 顶部集成了 PDF、DOCX、Word 等多个底层引擎，任何一个第三方包缺失都会导致进程在启动瞬间 `exit(1)`。
**修复指令**：
1. 激活环境：`conda activate roo_mcp`
2. 补齐依赖：在项目根目录执行 `pip install -r requirements.txt` 校验并补齐所有缺失依赖（`requirements.txt` 为依赖清单唯一权威源）。

### 异常表象 B：标准输出流污染 (Stdout 破坏)
**特征**：终端没有报错，但打印出了普通的中文/英文提示字符串（例如 `载入模型成功`、某些第三方库的初始化 log 等）。MCP 协议严格依赖 `stdio` 传输纯 JSON 数据，任何多余的 `print()` 都会直接导致客户端解析崩溃并掐断连接。
**修复指令**：
1. 全局检索 `tools/` 目录下的所有 `.py` 文件。
2. 找到所有暴露在全局作用域（`if __name__ == '__main__':` 之外）的 `print()` 语句。
3. 将其删除，或重定向到标准错误流：`print("log", file=sys.stderr)`。

### 异常表象 C：Windows 编码页陷阱
**特征**：终端执行后没有任何输出，光标正常悬停闪烁（处于等待 stdin 输入状态），但在 Kilo Code 中挂载依然报 `-32000`。
**原因研判**：Windows 默认终端编码（cp936/GBK）与 MCP 强制要求的 `utf-8` JSON 流发生编解码冲突。
**修复指令**：
1. 在 Windows 系统环境变量，或 Kilo Code 的 MCP 配置 JSON 中，强行注入环境变量：
   * `PYTHONIOENCODING=utf-8`
   * `PYTHONUTF8=1`
