# 风险评估 — patent_md_to_docx 专利 Markdown 逆向还原引擎

> **审计时间**: 2026-05-31 11:10 CST
> **审计版本**: v1.0 (项目立项前风险评估)
> **风险等级说明**: P0-致命 / P1-高危 / P2-隐患
> **基准报告**: 《项目现状梳理_20260531_1108_v1.0.md》

## 1. 工业级水位判定

基于联网查证与现有 `Roo_Code_utils` 基座对标，`patent_md_to_docx` 项目的预期水位评估如下：

| 维度 | 水位 | 评级 | 说明 |
|------|------|------|------|
| 核心算法能力 | **中高** | ⭐⭐⭐⭐ | 依赖外部库的 LaTeX → OMML 管道是业界通用路线，但 MML2OMML.XSL 的许可不确定性拉低了自主可控评分 |
| 与现有体系集成度 | **极高** | ⭐⭐⭐⭐⭐ | 完美继承现有 MCP 工具架构，python-docx / lxml / Pillow 均为现有依赖 |
| 安全防护 | **中** | ⭐⭐⭐ | 受益于现有 `_validate_input_file()` 防御层；但 XSLT 处理可能引入 XML 注入风险 |
| 可运维性 | **中** | ⭐⭐⭐ | 与现有工具平级，但公式转换失败时的降级日志需要特别设计 |
| 代码工程质量 (预期) | **中** | ⭐⭐⭐ | 如基于 `md_to_docx.py` 扩展，存在代码耦合和技术债务遗留风险 |
| 上游数据依赖性 | **低 (高风险)** | ⭐⭐ | 严重依赖 `docx_to_md.py` 的产出格式稳定性和正确性；上游的任何变更都会传导到本工具 |

**综合判定**: 项目在**技术可行性上不存在阻塞性问题**，但存在**一项架构级许可合规风险** (MML2OMML.XSL) 和**多项数据质量风险** (上游格式不稳定、转换保真度)。如能解决许可问题，项目可从 `md_to_docx.py` 快速演进到可用状态。

## 2. 核心风险点排序

### [P0-致命] 风险 — 可导致项目不可行或产生法律/安全灾难

#### P0-1: MML2OMML.XSL 许可合规与分发风险

- **位置**: 计划引入的外部 XSLT 文件 `MML2OMML.XSL`
- **来源**: Microsoft Office 安装目录 (`C:\Program Files\Microsoft Office\root\Office16\MML2OMML.XSL`)
- **风险本质**: 该文件是 **Microsoft Office 的私有组件**，不是独立开源项目。其许可条款未被 Microsoft 公开发布。将 XSL 文件纳入项目仓库或随工具分发构成**潜在的版权侵权**。
- **触发条件**: 项目公开发布到 GitHub、PyPI 或供第三方使用时，Microsoft 版权方可能发出 DMCA 下架通知。
- **后果**:
  - 项目无法在 PyPI / GitHub 上公开发布
  - 用户无法"开箱即用"——必须手动从 MS Office 安装目录复制文件
  - 降低了 AI Agent 自动化体验 (需要人工介入部署)
- **联网查证**: Stack Overflow 多个回答明确指出该文件存在于 MS Office 安装目录 ([链接](https://stackoverflow.com/questions/71051581))，但无任何地方确认其可自由再分发。
- **工业参考**: 开源项目 `tex_math_to_word` (GitHub: yeewantung) 同样面临此问题，其文档要求用户"自行安装 texmath (Haskell)"来绕过 MML2OMML.XSL 的直接分发。

**止血预案 (仅架构层面)**:
1. **方案 A (推荐)**: 改为手动构建 OMML 节点树 (`lxml` + `python-docx` OxmlElement)，完全绕过 MML2OMML.XSL。技术路线: 解析 MathML 的 `<mfrac>`, `<msup>`, `<msub>`, `<mrow>` 等元素 → 逐一映射为 OMML 的 `<m:f>`, `<m:sSup>`, `<m:sSub>`, `<m:r>` 等。这是**零外部依赖、许可清洁**的最高境界方案。
2. **方案 B (快速通路)**: 将 MML2OMML.XSL 作为"运行时自动检测"资源 — 启动时在 `C:\Program Files\Microsoft Office\root\Office16\` 自动发现；检测失败则降级到纯文本/PNG 模式。
3. **方案 C (退路)**: 使用 Pandoc 子进程 (`pandoc -f latex -t docx`)，Pandoc 使用 MIT 许可，包含自己的 OMML writer (不依赖 MS XSL)。

#### P0-2: LaTeX → OMML 双跳管道的数学校验失真

- **位置**: LaTeX → MathML (`latex2mathml`) → OMML (`MML2OMML.XSL`) 双向转换
- **风险本质**: 每增加一层转换，公式语义的保真度就损失一层。`latex2mathml` 仅覆盖约 80% 的 LaTeX 命令 (根据 GitHub Issues 反馈)，复杂结构 (`\begin{aligned}`, `\begin{pmatrix}`, `\xrightarrow`, `\overset`) 可能彻底失败或产生错误 MathML。
- **触发条件**: 用户编写了超出 `latex2mathml` 覆盖范围的 LaTeX 命令
- **后果**:
  - 公式在 DOCX 中显示为乱码、缺失或错误
  - 静默错误 —— 公式看起来正常但数学含义已改变 (最危险)
  - `MML2OMML.XSL` 两端的 XSLT 处理器 (`lxml` vs MS Office 内建) 输出可能不完全一致
- **工业参考**: Ecuaciones a Word (SaaS 工具) 采用"上传 → 服务器端 Pandoc 转换 → 返回 DOCX"模式来规避此问题；MarkView 采用三层降级 (OMML / PNG / Text)。[参考](https://www.ecuacionesaword.com/en/blog/markdown-latex-to-word-docx)

**止血预案 (仅架构层面)**:
1. 实现**三层降级策略**: Tier 1: OMML (最佳); Tier 2: PNG 图片渲染 (`matplotlib.mathtext`); Tier 3: 保留 LaTeX 原文字符串 (退化模式)
2. 增加**公式验证步骤**: 转换后检查关键符号 (如 `\frac` 是否生成 `<m:f>`) 是否存在，失败则自动降级
3. 提供 `formula_mode` 参数让用户显式选择管线，便于故障排查

### [P1-高危] 风险 — 可导致数据质量严重下降或功能不可用

#### P1-1: 上游产出物格式不稳定性 (docx_to_md 耦合风险)

- **位置**: `docx_to_md.py` 输出的 MD 格式变化传导到 `patent_md_to_docx.py`
- **风险本质**: `docx_to_md.py` 仍处于活跃迭代期 (存在死代码、逻辑重复、未调用的分流函数)，其 MD 产出格式 (公式包裹方式、图片路径模式、段落编号格式) 可能在后续版本中变化。
- **触发条件**:
  - `docx_to_md.py` 修改公式输出格式 (如从 `$$...$$` 变为 `\[...\]`)
  - `docx_to_md.py` 修改图片路径生成规则
  - 上游修复已知 bug 后改变了异常情况下的降级输出
- **后果**: `patent_md_to_docx.py` 的格式解析器失效，公式不被识别、图片路径找不到
- **缓解措施**: 在 `patent_md_to_docx.py` 中定义**明确的数据契约接口 (is-a)**——声明它期望的 MD 格式特征，并做格式自检测。
- **工业参考**: ["robustness principle" (Postel's Law) — "Be conservative in what you send, be liberal in what you accept"](https://en.wikipedia.org/wiki/Robustness_principle)

#### P1-2: 图片路径断裂与解析失败

- **位置**: MD 中对图片的引用 `![](.md/figures/rId34.png)`
- **风险本质**:
  - 路径 `rId34` 是 DOCX 内部关系 ID，对人类无意义，移机后无法追溯
  - 相对路径的基准点是 MD 文件所在目录，若 MD 被移动，所有图片引用断裂
  - 中文路径名 (`范加力-..._md/`) 可能在不同操作系统/文件系统编码下产生路径解析错误
- **触发条件**: MD 文件被移到其他目录；在 macOS/Linux 上处理 Windows 生成的 MD
- **后果**: DOCX 中图片位置显示为空白或错误占位符

**止血预案**:
1. 增加图片路径的多级回退搜索: 1) 严格匹配 → 2) 基于 MD 文件目录的相对解析 → 3) 基于 MD 文件名的 `*_md/figures/` 同级猜测
2. 对缺失图片生成占位符 (灰色矩形 + 文件名标注)，而非静默跳过

#### P1-3: 段落编号 `[NNNN]` 还原不全

- **位置**: 专利说明书中以 `[0001]`, `[0002]` 开头的段落
- **风险本质**: 原始 DOCX 中这些编号可能是自动编号列表 (Word numbering) 或手动文本前缀。`docx_to_md.py` 将其转换为**纯文本 `[NNNN]`**，反向转换时需要区分: (a) 作为编号恢复为 Word auto-numbering 还是 (b) 作为普通文本前缀附加。
- **后果**: 还原后的 DOCX 中段落编号不可被 Word 自动更新 (若用了手动编号)；或编号格式与 CNIPA 规范不一致

**止血预案**:
1. 采用"最小假设"策略 — 将 `[NNNN]` 作为带样式的文本前缀，不做自动编号 (避免编号序列错位)
2. 若未来需要自动编号，在 MD 中添加显式标记 (`<!-- auto-numbering -->`) 作为 opt-in

#### P1-4: 现有 `md_to_docx.py` 代码质量技术债务

- **位置**: `tools/md_to_docx.py` (150 行)
- **现状问题**:
  - 无模块化结构 — 单函数实现全部逻辑
  - 正则 `re.sub` 暴力 stripping 粗体/链接而非转换
  - 表格处理假设每行单元数一致，无容错
  - 无任何错误处理或日志
  - emoji 硬编码替换 (`✅` → `√`)
- **对 patent_md_to_docx 的影响**: 如果选择"扩展"而非"重写"，这些技术债务会直接继承
- **建议**: 重写为模块化结构，将公共工具函数 (字体设置、段落创建、表格构建) 抽取为共享模块

#### P1-5: 无 Diff 验证机制 — 无法量化还原度

- **风险本质**: DOCX → MD → DOCX 闭环转换后，无法自动量化"还原的 DOCX 与原始 DOCX 的差异"。当前仅能人工肉眼比对，效率低且不可靠。
- **后果**: 无法建立对转换质量的信心；bug 可能长期潜伏
- **工业参考**: Docling、MinerU 等项目使用"结构化对比 + BLEU 分数"衡量 DOCX 解析质量

**止血预案**:
1. 建立**回归测试集**: 选 3-5 个典型专利 DOCX → DOCX→MD→DOCX 闭环
2. 解压两个 DOCX 的 `word/document.xml`，用 `xmldiff` 或自定义 XPath 比对关键节点 (段落数、公式数、图片数)
3. 不追求逐字节一致，只验证关键结构元素数量一致

### [P2-隐患] 风险 — 可导致边际体验下降或长期维护成本上升

#### P2-1: olefile 遗留依赖维护停滞

- **位置**: `tools/mtef_parser.py` 和 `tools/mtef_fast.py` 依赖 `olefile`
- **风险**: `olefile` 自 2018 年停止活跃维护 (v0.47)。虽然本工具不直接使用 MTEF 解析功能 (patent_md_to_docx 不处理二进制公式)，但如果未来需要"直接生成 MathType OLE 公式 (而非 OMML)"这一高级功能，将受限于 olefile 的维护状态。
- **评级**: P2 — 公式写入功能属于远期需求，当前不阻塞。

#### P2-2: 中文专利 CNIPA 格式规范变更风险

- **风险**: CNIPA 自 2025 年 10 月 1 日起要求所有电子申请使用 XML 格式（非 DOCX）。虽然 DOCX 仍在事务所内部流转中使用，但如果 CNIPA 完全弃用 DOCX 格式，本工具的"还原 DOCX"需求可能随时间推移而减弱。
- **评估**: 即使 CNIPA 官方不使用 DOCX，律师事务所和发明人之间的内部校对、修改流程中 DOCX 仍将长期存在。本工具的目标场景是"AI Agent 辅助修改专利 → 输出 DOCX 供人工最终审定"，这一需求不受 CNIPA 受理格式变更影响。

#### P2-3: `latex2mathml` 库的功能覆盖边界

- **覆盖情况**: `latex2mathml` 支持常见数学符号、分数、上下标、根号、求和/积分、希腊字母、矩阵 (部分)。不支持: `\begin{aligned}` 多行对齐、`\xrightarrow` 等可伸缩箭头、`\overset`/`\underset`、`\substack`、`\cancel` 等。
- **对专利文档的影响**: 专利文档中的公式通常相对简单 (主要是下标/上标/希腊字母/分数)，大部分可被 `latex2mathml` 覆盖。矩阵和对齐环境在工程专利中较少出现。
- **评级**: P2 — 实际使用中影响面有限，但需在产品文档中明确声明支持范围。

#### P2-4: XSLT 安全风险 (XML 外部实体注入 XXE)

- **位置**: `lxml.XSLT` 处理外部 MathML 输入
- **风险**: 如果 `latex2mathml` 生成的 MathML 中包含外部实体引用 (`<!ENTITY>`)，`lxml` 默认配置可能允许实体解析，导致文件泄露或 SSRF。
- **缓解**: `lxml.etree.XSLT` 默认禁用外部实体 (与 `etree.XMLParser` 不同)，风险较低但应在代码中显式确认。
- **评级**: P2 — lxml 默认安全性较好，但建议增加显式的 XML 解析安全配置。

## 5. 架构决策建议矩阵

| 决策项 | 推荐 | 理由 |
|--------|------|------|
| 复用 vs 重写 md_to_docx.py | **模块化重写** | 现有代码质量不足以承受专利文档的复杂度，且重写成本 (150 行) 可控 |
| LaTeX → OMML 路线 | **方案 A (手动 OMML) 为长期目标，方案 B (MML2OMML.XSL 运行时检测) 为快速通路** | 兼顾快速交付与长期许可清洁 |
| 代码结构 | **独立 `tools/patent_md_to_docx.py`** + 抽取公共工具函数到 `tools/docx_utils.py` | 避免与通用 md_to_docx 耦合，同时复用字体设置等公共逻辑 |
| 公式降级策略 | **三层: OMML → PNG → Text** | 最坏情况至少保留"可读文本"，不丢数据 |
| MCP 集成 | **新增独立 tool `patent_md_to_docx`** (非替换 `convert_md_to_docx`) | 两工具受众不同，通用工具保持简洁，专利工具专业深化 |
